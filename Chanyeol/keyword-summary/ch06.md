- JPA란?

  Java Persistence API, 자바 진영의 ORM 기술 표준 인터페이스

  ORM: 자바 객체와 관계형 DB 테이블을 자동으로 매핑해주는 기술

  개발자가 직접 SQL을 짜지 않아도 JPA가 자동으로 쿼리 생성 및 실행 → 반복적인 CRUD 작업 줄여줌

- N+1 문제란?

  쿼리 1번으로 N개의 데이터를 가져왔는데, 각 데이터의 연관 엔티티를 조회하려고 추가로 N번의 쿼리가 더 발생하는 문제

  데이터 양이 많을수록 쿼리 수가 많아져 데이터베이스에 심각한 부하를 일으킴

  해결: **지연 로딩(FetchType.LAZY)를 사용하여 위험도를 낮춘다.**

- 지연로딩과 즉시로딩의 차이는?
    - 즉시로딩
        - 연관 엔티티를 즉시 함께 조회
        - JOIN해서 한 번에 쿼리 발생
        - N+1 문제 위험도 높음
    - 지연로딩
        - 실제 사용 시점에 엔티티 조회
        - 필요할 때 마다 쿼리 발생(사용하지 않으면 쿼리 발생 x)
        - N+1 문제 위험도 낮음
- JPQL란?

  Java Persistence Query Language, 테이블이 아닌 엔티티 객체를 대상으로 작성하는 쿼리 언어

  SQL의 테이블명 대신 클래스명, 컬럼명 대신 필드명 사용

    ```c
    // SQL
    SELECT * FROM member WHERE name = '백구';
    
    // JPQL
    SELECT m FROM Member m WHERE m.name = '백구'
    ```

- Fetch Join란?

  JPQL에서 N+1 문제를 해결하기 위해 연관 엔티티를 한 번의 쿼리로 함께 조회하는 방법

  루프에서 연관 엔티티 접근 시, 응답 DTO에 연관 엔티티 필드가 포함될 때, 단건 조회인데 연관 엔티티도 함께 필요할 때 사용한다.

  보통은 지연로딩을 기본으로 깔고 연관 엔티티 데이터가 필요한 경우에 쓴다고 보면 된다고 한다.

- @EntityGraph란?

  N+1 문제를 해결하는 또 다른 방법, JPQL 없이 어노테이션으로 선언 가능

    ```c
    // ("select m from Member m left join fetch m.team" 쿼리와 똑같은 결과)
    @Override
    @EntityGraph(attributePaths = {"team"}) // 옵션 : 연관된 엔티티 지정
    List<Member> findAll();
    ```

  로딩할 필드명을 지정하면, 해당 연관 엔티티를 EAGER처럼 한 번에 가져온다.

- commit과 flush 차이점은?
    - commit
        - flush 후 트랜잭션을 DB에 바로 최종 반영
        - 트랜잭션이 종료되어 롤백 불가능
        - 영속성 컨텍스트 종료
    - flush
        - 영속성 컨텍스트의 변경사항 DB에 SQL 전송
        - 롤백 가능
        - 영속성 컨텍스트 유지

  ❗️트랜잭션: 여러 작업을 하나의 묶음으로 처리하는 단위

  ❗️영속성 컨텍스트: JPA가 엔티티를 관리하는 메모리 공간(1차 캐시)

  트랜잭션 시작
  ↓
  영속성 컨텍스트 활성화 (엔티티 캐시 시작)
  ↓
  flush → SQL만 DB로 전송 (캐시 유지, 롤백 가능)
  ↓
  commit → 최종 반영 + 영속성 컨텍스트 초기화 (롤백 불가)
  ↓
  트랜잭션 종료
