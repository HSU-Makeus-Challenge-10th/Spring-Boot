## 빌더패턴이란?

> **빌더 패턴 Builder Pattern**
> 객체의 생성 과정과 표현 방법을 분리하여 동일한 생성 절차에서 서로 다른 표현 결과를 만들 수 있게 하는 생성형 디자인 패턴
> → 객체를 생성할 때 필요한 데이터들을 순차적으로 입력받아 최종적으로 안전한 형태의 객체를 완성해 내는 방법
>
> **장점**
> * 가독성: 메서드 체이닝 방식을 사용하기에 각 인자가 어떤 의미인지 직관적인 파악이 가능함
> * 유연성: 선택적 매개변수를 해당 메서드로 처리 가능
> * 불변성과 안전성 보장: `Setter` 없이도 객체를 초기화가 가능 → 한 번 생성된 객체의 데이터가 외부에서 변경되는 것을 막을 수 있다는 이점이 있음
>
> **단점**
> * 보일러플레이트 코드 증가: 변수 선언의 중복성으로 인해 코드 증가 문제가 있음


## record vs static class

| 구분 | public static class | record |
| :--- | :--- | :--- |
| **특징** | **가변성 (Mutable)**<br>- Lombok `@Setter`를 통해 값 변경 가능<br>- 불변(immutable)하게 만들려면 `final` 키워드와 `@Value` 등을 사용해야 함 | **불변성 (Immutable)**<br>- 선언과 동시에 모든 필드가 `private final`이 됨<br>- Setter가 원천적으로 없음 |
| **자동 생성** | 어노테이션 기반으로 생성 | Java 컴파일러가 네이티브로 생성 |
| **접근자** | JavaBean 규약: `getName()` | 필드명 동일: `name()` |
| **값 변경 방법** | Setter 사용 | 변경 불가!<br>필요시 새 객체를 생성해야 함 |

<br>

| 구분 | public static class | record |
| :--- | :--- | :--- |
| **장점** | - 유연성 : `Setter`를 사용할 수 있어 객체 생성 후에도 값을 변경할 수 있음<br>- 범용성 : Java 8, 11 등 레거시 프로젝트에서도 사용할 수 있는 표준 방식<br>- 프레임워크 호환성 : 오래된 라이브러리나 프레임워크가 JavaBean 규약(`getName()`, 기본 생성자)을 요구할 때 문제가 없음 | - 간결성 : DTO의 정의를 한 줄로 끝낼 수 있어 가독성에 좋음<br>- 강제적 불변성 : 데이터는 불변이어야 한다는 DTO의 본질적인 목적을 Java 문법 자체로 보장함 (Thread-safe)<br>- 의도 명확성 : `record` 키워드 자체 |
| **단점** | - 코드량 : `record`에 비해 코드가 길어짐<br>- 불변성 보장 : 불변 객체로 만들려면 `@Value`를 쓰거나 `final`을 일일이 붙이는 등 효율성 또한 떨어짐 | - Java 버전 제한 : Java 16 이상에서만 정식으로 사용할 수 있습니다.<br>- 유연성 부족 : Setter가 필요한 경우 |


## 제네릭이란?

> **제네릭 Generics**
> 데이터의 타입을 미리 고정하지 않고 외부에서 객체를 생성하거나 메서드를 호출할 때 지정하도록 파라미터화하는 문법으로 `<T>`, `<E>`, `<V>` 같은 기호로 표현
>
> **장점**
> * 타입 안정성: 의도하지 않은 타입의 데이터가 들어가는 것을 컴파일에 막아주기에 런타임 에러를 방지할 수 있음
> * 형변환 생략 가능: 데이터를 꺼낼 때마다 캐스팅할 필요가 없음
> * 코드 재사용성 증가: 여러 타입에 대해 재사용이 가능함
>
> **단점**
> * 기본 데이터 타입 사용 불가: `<int>`, `<double>`, `<boolean>` 같은 기본 타입은 제네릭 타입 파라미터로 넣을 수 없고 반드시 객체 형태인 `<Integer>`, `<Double>`, `<Boolean>` 같은 래퍼 클래스를 사용해야 됨
> * 타입 소거: 런타임에 제네릭 타입 정보가 지워짐 → 런타임에 제네릭 타입을 검사하는 코드 작성이 불가함


## @RestControllerAdvice이란?

> **@RestControllerAdvice**
> 스프링 프레임워크에서 RESTful API의 전역 예외 처리를 위한 어노테이션
>
> `@RestControllerAdvice`가 없을 경우 .. 모든 컨트롤러의 모든 메서드 안에서 `try-catch`로 예외를 직접 처리해야
