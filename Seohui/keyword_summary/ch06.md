## 1. JPA란?
**JPA(Java Persistence API)**
ORM 기술을 어떻게 사용해야 하는지에 대한 규칙 및 가이드라인을 정의한 명세서로, JPA를 실제로 구현하기 위해서는 Hibernate 와 같은 ORM 프레임워크가 필요하다

### JPA 동작
* **엔티티(Entity)**: 데이터베이스의 테이블에 매핑되는 자바 클래스로 `@Entity` 어노테이션을 붙여서 JPA에 알림. 엔티티 객체를 중심으로 코드 작성이 이뤄짐
* **EM(EntityManager)**: 엔티티를 관리하는 객체로, 엔티티를 데이터베이스에 저장(persist), 조회(find), 수정(merge), 삭제(remove)하는 모든 CRUD 작업을 EM으로 수행한다

### JPA 장점
* **생산성 향상**: 반복적인 코드를 직접 작성할 필요 없이 객체를 중심으로 개발에 집중할 수 있어 개발 속도를 높여줌
* **유지보수 용이**: SQL 쿼리가 코드에서 분리됨 → 비즈니스 로직이 더 명확해짐. 테이블 컬럼이 하나 추가되더라도 엔티티 클래스의 필드 하나만 추가하면 됨 → 유지보수 측면에서도 편리해진다
* **데이터베이스 독립성**: 특정 데이터베이스에 종속되지 않음. JPA 코드는 그대로 두고 설정 파일에서 수정하면 MySQL을 PostgreSQL이나 Oracle로 쉽게 DB 변경이 가능해짐
* **성능 최적화**: 트랜잭션 단위로 SQL을 모아서 한 번에 전송하는 다양한 성능 최적화 기능을 지원해줌

## 2. N+1 문제란?
**N+1 문제**
연관 관계에서 발생하는 이슈로, 연관 관계가 설정된 엔터티를 조회할 경우에 조회된 데이터 갯수(n) 만큼 연관관계의 조회 쿼리가 추가로 발생하여 데이터를 읽어옴

### FetchType
연관관계를 갖고 있는 객체를 조회하면 JPA는 해당 객체가 들고 있는 연관관계의 객체들을 모두 조회에서 끌고옴
→ **이때 1+N 쿼리 문제 발생**

* **FetchType.EAGER (즉시로딩)**: 연관된 객체를 즉시 가져오는 방식. 연관된 객체가 사용되지 않더라도 일단 모두 조회해옴
* **FetchType.LAZY (지연로딩)**: 연관객체를 프록시 객체로 대체하여 껍데기 객체로 대치하여 추후 직접적인 조회가 발생하면 가져오는 방식
    * **프록시(Proxy) 객체**: 대리자, 대신 처리해주는 객체 → 가짜 객체 (조회를 미루는 것)

> 그럼에도 지연로딩으로 N+1 문제를 해결할 수는 없음..

### N+1 문제 해결 방법
1. **Fetch Join**: `fetch join`을 사용하면 즉시 한 번의 쿼리로 연관 엔티티를 모두 조회함 (단, 컬렉션 페치 조인은 페이징 불가 → 데이터 정확성이 깨질 수 있음)
2. **EntityGraph (Spring Data JPA)**: Fetch Join을 대신하여 간편하게 사용 가능. 동적 쿼리에는 `@EntityGraph` + JPQL로도 가능
3. **BatchSize 설정**: JPA는 `@OneToMany`에서 LAZY 로딩을 하면서 일괄적으로 가져올 수 있게 하는 설정

## 3. 지연로딩과 즉시로딩의 차이는?
* **지연로딩 (FetchType.LAZY)**
    * 필요한 시점에 연관된 객체의 데이터를 불러오는 방식
    * 필요할 때 연관된 엔터티를 DB에서 조회
    * 처음에는 프록시 객체만 있음. 진짜 데이터는 나중에
    * `@OneToMany` 에서는 기본이 LAZY
* **즉시로딩 (FetchType.EAGER)**
    * 필요와 무관하게 연관되는 모든 객체의 데이터까지 한 번에 불러오는 방식
    * 엔터티를 조회할 때 연관 객체도 같이 로딩
    * 조인 쿼리로 한번에 가져와서 N + 1 문제 발생

## 4. JPQL이란?
**JPQL (Java Persistence Query Language)**
자바 객체를 대상으로 하는 SQL을 추상화한 객체 지향 쿼리 언어

### 사용 이유
SQL 은 DB 구조에 의존적인 반면에 JPQL 은 자바 객체(엔터티)를 기준으로 쿼리를 짤 수 있어 DB에 독립적인 코드가 가능함

### SQL vs JPQL 비교
| 구분 | SQL | JPQL |
|---|---|---|
| **대상** | 테이블 | 엔티티 클래스 |
| **컬럼 이름** | DB 컬럼명 | 자바 필드명 |
| **테이블 이름** | DB 테이블명 | 자바 클래스명 |

## 5. Fetch Join이란?
**Fetch Join**
지연로딩된 연관관계의 객체를 한 번의 쿼리로 같이 가져오는 JPQL 문

### 사용 이유
JPA는 기본적으로 연관 엔티티를 지연로딩(LAZY) 처리
→ 연관 데이터를 반복 접근하면 N+1 문제 발생
→ Fetch Join 을 사용해 한 번의 쿼리로 전부 로딩

```sql
select distinct u from User u join fetch u.reviews
