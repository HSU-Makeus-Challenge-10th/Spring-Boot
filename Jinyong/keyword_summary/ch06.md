- JPA란?

  JPA(Java Persistence API)란, 자바 진영에서 ORM(Object-Relational Mapping) 기술의 표준으로 사용되는 인터페이스의 모음이다. → **Java에서 ORM을 어떻게 쓰는지에 대한 규칙**이라고 생각하면 된다.

  (여기서 ORM은 애플리케이션의 클래스와 RDB(Relational DataBase)의 테이블을 매핑(연결)한다는 뜻)

  원래는 DB를 사용할려면 SQL을 직접 작성해야 했지만, JPA를 통한다면 Java와 관계형 데이터 베이스 사이의 차이를 줄여준다. (DB와 Java는 각각 지향하는 점이 다르기에 개발 과정에서 설계가 달라질 수 밖에 없음 → 객체-관계 불일치 문제)

  JPA를 사용하면서 얻게 되는 장점은,

    1. SQL 반복 작성이 줄어들게 되고(INSERT, SELECT, DELETE, UPDATE를 직접 하나하나 작성 안 해도 된다.)
    2. 객체 중심 개발이 가능해진다. → 코드로 도메인 객체 생성, 객체 관계를 코드로 표현 가능해진다.

  반대로 단점은,

    1. 복잡한 쿼리는 직접 작성해야 할 때가 있다.
    - 기본적인 CRUD에는 좋지만, 복잡한 조인 쿼리는 JPQL이나 다른 것들을 이용해 쓸 수 있다.

- N+1 문제란?

  N+1문제란 JPA(ORM)에서 연관관계가 있는 엔티티를 조회할 때, 처음 실행한 쿼리 1번에 더해 조회된 데이터 개수 N개만큼 연관된 엔티티를 조회하는 쿼리가 추가로 발생하는 현상으로 성능이 저하되는 문제를 말한다.

  Ex) - 회원 목록 하나를 조회(1 조회)하는데 각 회원의 리뷰까지 조회(회원의 수 N만큼 더해짐)되는 경우

  N+1 문제가 발생하면 쿼리의 수가 많아져 DB의 부하와 응답 속도 저하가 발생할 수 있다.

  **why? 왜 발생할까?**

    - 문제는 연관관계 조회 방식에 있다. (이건 다음 토글에서 계속)

- 지연로딩과 즉시로딩의 차이는?


    | 방식 | 의미 |
    | --- | --- |
    | 지연 로딩(**FetchType.LAZY**) | 연관 데이터를 실제로 사용할 때 조회(요청이 오면 그때 그때) |
    | 즉시 로딩
    (**FetchType.EAGER**) | 처음 entity를 조회할 때 연관 데이터도 함께 조회(한꺼번에 조회) |
    - 지연 로딩에서는 연관 데이터를 바로 가져오지 않고, 실제로 접근할 때만 가져오는 방식으로 만약 회원 목록을 조회하면 회원 목록만! 조회하고 리뷰는 조회하지 않는다. (이후 리뷰 조회를 호출하면 그때 리뷰를 조회한다)
    
    지연 로딩은 불필요한 데이터 조회를 줄일 수 있다는 장점이 있지만, 반복문 안에서 연관 데이터에 접근하면 N+1 문제가 발생할 수 있다.
    
    - 즉시 로딩에서는 처음 엔티티를 조회할 때 연관 데이터도 함께 가져오는 방식이다. 만약 회원 목록을 조회하면 회원 목록과 관련된 리뷰 조회까지도 데이터를 가져온다.
    
    즉시 로딩은 필요한 데이터를 미리 가져올 수 있지만, 항상 필요하지 않은 데이터까지 조회할 수 있고 상황에 따라 즉시 로딩에서도 N+1 문제가 발생할 수 있다
    
    이런 차이 때문에 일반적으론 지연 로딩을 지향하고 연관 데이터가 반드시 필요한 경우에 즉시 로딩을 사용하는 방식으로 많이 사용한다. 

- JPQL란?

  *“JPA의 일부로, 쿼리를 데이터베이스의 테이블이 아닌 JPA 엔티티, 즉 ’객체‘를 대상으로 작성하는 객체 지향 쿼리 언어입니다.”*

  *“즉, SQL처럼 데이터베이스와 상호 작용한다는 기본 컨셉은 가져가면서, 객체 지향적인 특성을 반영한 쿼리를 날릴 수 있게 도와주는 것이 JPQL이라고 보시면 됩니다”* - (워크북 발췌)

  SQL은 테이블을 대상으로 쿼리를 작성한다면, JPQL은 JPA의 엔티티 객체를 대상으로 쿼리를 작성한다.  → 결국 **무엇을 대상으로 쿼리를 작성**하냐의 차이

  JPQL을 사용하는 이유는

  JPA의 기본 메서드만으로도 간단한 조회가 충분히 가능하지만,  만약 복잡한 조회가 필요할 때에는 직접 작성을 통한 쿼리를 작성 할 수 있다는 장점이 있다. (Ex. `@Query(”select m from Member m”)` )

  ⇒ **JPA에서 복잡한 조회 조건을 객체 중심으로 작성하니 위한 쿼리 언어이다!**

  알아야 할 점!

    - JPQL은 결국 SQL로 변환되어서(by Hibernate) DB에 보내진다 → DB는 JPQL을 직접 이해 불가능
    - JPQL은 SQL과 비슷해 보여도 테이블명과 컬럼명이 아니라 엔티티명과 필드명을 사용해야 하며, 실제 실행되는 SQL을 확인하지 않으면 성능 문제를 놓칠 수 있다.

- Fetch Join란?

  **JPQL에서 연관된 엔티티를 한 번에 함께 조회하기 위해 사용하는 기능**이다.
  일반 Join이 단순히 비교 혹은 필터링을 위한 것이라면 Fetch Join은 연관된 엔티티를 실제 객체로 같이 가져오는 것이 목적인 기능이다.

  **why? 왜 사용하는 것일까?**

  **N+1 문제를 줄이기 위해서이다.**

  예를 들어 회원과 리뷰가 연관관계에 있을 때, 회원 목록을 조회한 뒤 각 회원의 리뷰까지 사용해야 한다면 일반적인 지연 로딩에서는 회원 수만큼 리뷰 조회 쿼리가 추가로 발생할 수 있다.

  이때 Fetch Join을 사용하면 회원과 리뷰를 한 번의 조인 쿼리로 함께 조회할 수 있다.

  → **여러 번 나갈 수 있는 연관 엔티티 조회 쿼리를 Join을 이용해 한 번에 가져오는 방식**이기 때문!

  연관 엔티티를 처음 조회할 때 함께 가져오기 때문에, 반복문에서 연관 데이터에 접근하더라도 추가 쿼리가 계속 발생하는 것을 줄일 수 있다.

  주의해야 할 점!

  N+1 문제를 해결하는 데 도움이 되지만, 무조건 좋은 것은 아니다.

  회원 목록만 필요한 상황에서 리뷰까지 Fetch Join으로 가져오면 쿼리 수는 1번일 수 있지만, 필요 없는 데이터까지 가져올 수도 있다. 이것은 비단 **N+1 문제** 뿐만 아니라 **과도한 조회 문제가 될 수 있다.**

  따라서 Fetch Join은 연관 데이터가 실제로 필요한 조회에서만 선택적으로 사용해야 한다.

- @EntityGraph란?

  @EntityGraph는 JPA에서 특정 조회를 실행할 때 함께 조회할 연관 엔티티를 지정하는 기능이다.

    - 엔티티의 기본 로딩 전략이 `LAZY`로 되어 있더라도, 특정 Repository 메서드에서만 연관 데이터를 함께 가져오도록 설정할 수 있다.

  **why? 왜 사용할까?**
  이 기능 또한 Fetch Join과 마찬가지로 **N+1 문제를 줄이기 위해서이다.**

  @EntityGraph는 JPQL에 직접 join fetch를 적지 않고도 특정 연관관계를 함께 조회하도록 지정하는 방법이다.

  둘의 차이점 이렇다.

  | 구분 | Fetch Join | @EntityGraph |
      | --- | --- | --- |
  | 사용 위치 | JPQL 안에서 작성 | Repository 메서드 위에 어노테이션으로 작성 |
  | 예시 | join fetch m.reviews | @EntityGraph(attributePaths = {"reviews"}) |
  | 목적 | 연관 엔티티를 함께 조회 | 연관 엔티티를 함께 조회 |
  | 장점 | 쿼리에서 조인 대상을 명확하게 제어 가능 | 기존 메서드에 간단히 적용 가능 |
  | 단점 | JPQL을 직접 작성해야 함 | 복잡한 조건 조인 제어는 Fetch Join보다 제한적일 수 있음 |

  주의해야 할 점!

    - `@EntityGraph`도 Fetch Join과 마찬가지로 **필요한 경우에만 선택적으로 사용해야 한다.**

  회원 목록만 필요한데 리뷰까지 함께 조회하면 N+1 문제는 아니지만, 필요 없는 데이터를 가져오는 **과도한 조회**가 될 수 있다.(Fetch Join과 같은 이유!)

  또한 `@EntityGraph`는 연관 엔티티를 함께 조회하는 데 편리하지만, 복잡한 조건이나 세밀한 조인 제어가 필요한 경우에는 JPQL의 Fetch Join이 더 적합할 수 있다.

- commit과 flush 차이점은?
    - commit은 트랜잭션을 최종 확정하여 DB 변경 사항을 영구 반영하는 과정이다.
    - flush는 영속성 컨텍스트의 변경 내용을 DB에 SQL로 보내는 과정이다.

  | 구분 | commit | flush |
      | --- | --- | --- |
  | 의미 | 트랜잭션을 최종 확정 | 영속성 컨텍스트의 변경 내용을 DB에 SQL로 동기화 |
  | 실행 결과 | DB 변경 사항 영구 반영 | SQL이 DB에 전달됨 |
  | 트랜잭션 종료 여부 | 종료됨 | 종료되지 않음 |
  | rollback(되돌리기)가능 여부 | commit 후에는 일반적으로 불가 | commit 전이면 rollback 가능 |
  | 자동 실행 시점 | 트랜잭션 정상 종료 시 | commit 직전, 또는 쿼리 실행 전 등 |
  | 핵심 역할 | 변경 내용 확정짓기 | 변경 내용을 DB에 보내기 |

  **여기서 트랜잭션(Transaction)은 데이터베이스 작업을 하나의 작업 단위로 묶은 것**이다.

  DB 작업 여러 개를 **“전부 성공하면 반영하고, 하나라도 실패하면 전부 취소하는 묶음”**이라고 보면 된다.

  commit과 flush는 모두 DB 반영과 관련된 개념이지만 역할이 다르다.

  flush는 영속성 컨텍스트의 변경내용을 SQL로 만들어 DB에 동기화하는 과정이고,

  commit은 트랜잭션을 최종 확정하여 변경 사항을 영구 반영하는 과정이다.

  일반적으론 JPA에서 commit직전 flush가 발생하지만, flush가 실행되서 SQL이 DB에 전달되도 결국 commit이 이뤄지지 않았다면, 반영이 안된 것이다.

  즉 flush는 “DB에 보내기”, commit은 “작업 최종 확정짓기”라고 생각하면 된다.
