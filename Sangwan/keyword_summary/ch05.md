- 빌더패턴이란?

  객체의 **생성 과정과 표현 방법을 분리**하여, 복잡한 객체를 단계별로 생성할 수 있게 돕는 생성 디자인 패턴
    
  ---

  #### 등장 배경

    1. 점진적 생성자 문제 (Telescoping Constructor)

  매개변수가 많아질수록 생성자의 인자 순서를 파악하기 어렵고, 실수로 값이 뒤바뀔 위험이 큽니다.

    ```java
    // 어떤 인자가 무엇을 의미하는지 한눈에 들어오지 않음
    User user = new User("gildong", 25, "Seoul", "010-1234-5678", "test@email.com");
    ```

    2. 자바빈즈 패턴의 불안정성

  Setter를 사용하면 객체 생성 후에도 값이 변경될 수 있어 **불변성(Immutability)** 보장이 어렵고, 생성 도중에는 객체가 불완전한 상태에 놓일 수 있습니다.
    
  ---

  #### 장점

    - **가독성 향상:** 메서드 체이닝을 통해 어떤 필드에 어떤 값이 들어가는지 명확히 알 수 있습니다.
    - **유연성:** 필수 매개변수와 선택적 매개변수를 자유롭게 조합하여 객체를 생성할 수 있습니다.
    - **객체 불변성 유지:** 생성 완료 후 내부 상태를 변경할 수 없도록 설계하기 용이합니다. (Setter 지양 가능)
    - **유효성 검사:** `build()` 메서드 호출 시점에 객체의 무결성을 검증할 수 있습니다.

    ---

  #### Spring Boot (Lombok) 실전 예제

  스프링 부트 환경에서는 Lombok의 `@Builder` 어노테이션을 사용하여 보일러플레이트 코드 없이 간결하게 구현하는 것이 관례입니다.

  #### 1. Entity 또는 DTO 정의

    ```java
    import lombok.Builder;
    import lombok.Getter;
    
    @Getter
    @Builder // 빌더 패턴 자동 생성
    public class MemberResponse {
        private final String loginId;   // 필수
        private final String nickname;  // 필수
        private final int age;          // 선택
        private final String email;     // 선택
    }
    ```

  #### 2. 객체 생성 및 활용

    ```java
    @Service
    public class MemberService {
        public void createMember() {
            // 체이닝 방식을 통해 가독성 있게 객체 생성
            MemberResponse member = MemberResponse.builder()
                    .loginId("hansung_user")
                    .nickname("코딩왕")
                    .age(24)
                    .email("it@example.com")
                    .build();
    
            // 필요한 데이터만 골라서 생성 가능 (순서 상관 없음)
            MemberResponse simpleMember = MemberResponse.builder()
                    .loginId("admin")
                    .nickname("관리자")
                    .build();
        }
    }
    ```
    
  ---

    - **언제 쓸까?** 생성자 인자가 4개 이상이거나, 객체의 불변성을 보장해야 하는 비즈니스 로직에서 필수적으로 권장됩니다.
    - **Spring Data JPA:** 엔티티 생성 시 `@Builder`를 활용하면 테스트 코드 작성 시 가독성이 비약적으로 상승합니다.
    - **주의:** 단순한 구조의 객체에 남용하면 클래스 수가 늘어나고 코드가 복잡해질 수 있으므로 상황에 맞게 적용합니다.
- record vs static class

  ### 1. static class (전통적인 방식)

  기존 자바에서 내부 클래스를 정의할 때 사용하는 표준적인 방법입니다.

    - **장점:** * **유연성:** 모든 라이브러리(QueryDSL, JPA 등)와 호환성이 완벽합니다.
        - **수정 가능:** 필요에 따라 필드 값을 변경할 수 있습니다.
    - **단점:** * **보일러플레이트:** `Getter`, `Setter`, `equals`, `hashCode`, `toString` 등을 일일이 구현하거나 Lombok(@Getter, @NoArgsConstructor 등)에 의존해야 합니다.
        - **가독성:** 코드가 길어지면 구조를 한눈에 파악하기 어렵습니다.
    - Lombok을 활용해도 여러 어노테이션이 필요합니다.

    ```java
    public class MemberRequest {
    
        @Getter
        @NoArgsConstructor
        public static class Create {
            private String name;
            private String email;
    
            @Builder
            public Create(String name, String email) {
                this.name = name;
                this.email = email;
            }
        }
    }
    ```
    
  ---

  ### 2. Java Record (최신 방식)

  Java 16부터 정식 도입된 데이터 중심 클래스로, 불변(Immutable) 데이터를 다루는 데 최적화되어 있습니다.

    - **장점:**
        - **간결함:** 필드만 선언하면 생성자, Getter, `equals`, `hashCode`, `toString`이 자동으로 생성됩니다.
        - **불변성:** 모든 필드가 `final`로 선언되어 데이터의 일관성을 보장합니다.
    - **단점:**
        - **제약 사항:** 상속이 불가능하며, 한 번 생성된 객체의 값을 수정할 수 없습니다.
        - **호환성 이슈:** 일부 라이브러리(QueryDSL 등)에서 Reflection이나 Proxy 생성 시 추가 설정이 필요하거나 사용이 까다로울 수 있습니다.
    - 어노테이션 없이 한 줄로 정의가 가능합니다.

    ```java
    public class MemberRequest {
    
        public record Create(
            String name,
            String email
        ) {}
    }
    ```

  | **구분** | **static class (+Lombok)** | **Java Record** |
      | --- | --- | --- |
  | **선언 코드** | 길고 복잡함 | 매우 간결함 |
  | **불변성** | 추가 설정 필요 (`@Value` 등) | 기본적으로 불변 (`final`) |
  | **데이터 접근** | `getName()` | `name()` |
  | **라이브러리 호환성** | 매우 높음 (QueryDSL, JPA 등) | 라이브러리에 따라 다름 |
  | **수정 가능 여부** | 가능 (Setter 사용 시) | 불가능 |
- 제네릭이란?

  클래스나 메서드에서 사용할 **데이터 타입을 외부에서 지정**하는 기법

  #### 1. 컴파일 타임 타입 체크

  제네릭을 사용하지 않으면 `Object` 타입을 사용한 뒤 매번 형변환(Casting)을 해야 합니다. 이 과정에서 실수로 잘못된 타입을 넣으면 런타임 에러(ClassCastException)가 발생할 수 있습니다. 제네릭은 이를 **컴파일 단계**에서 잡아줍니다.

  #### 2. 코드 재사용성

  하나의 클래스나 메서드로 다양한 타입(String, Integer, Custom Object 등)을 처리할 수 있어 중복 코드를 줄여줍니다.

    ```java
    @Getter
    @AllArgsConstructor
    public class ApiResponse<T> {
        private String status;
        private String message;
        private T data; // 어떤 객체든 들어올 수 있음
    
        public static <T> ApiResponse<T> success(T data) {
            return new ApiResponse<>("SUCCESS", "요청에 성공했습니다.", data);
        }
    }
    
    // 사용 예시
    ApiResponse<MemberResponse> response = ApiResponse.success(memberDto);
    ```

  #### 주요 용어 관례

  제네릭에서 사용하는 알파벳은 관례적으로 다음과 같은 의미를 가집니다.

    - **`<T>`**: Type (일반적인 타입)
    - **`<E>`**: Element (리스트 등 컬렉션의 요소)
    - **`<K>`**: Key (맵의 키)
    - **`<V>`**: Value (맵의 값)
    - **`<N>`**: Number (숫자 타입)
- @RestControllerAdvice이란?

  스프링 부트 애플리케이션에서 **전역적으로 예외를 처리**하기 위해 사용하는 어노테이션

  **개념 및 역할**

    - **@ControllerAdvice + @ResponseBody:** `@ControllerAdvice`의 기능을 수행하면서, 반환 값을 자동으로 JSON/XML과 같은 응답 본문(Response Body)으로 변환해 줍니다. REST API를 만들 때 필수적인 도구입니다.
    - **AOP(Aspect Oriented Programming):** 특정 패키지나 컨트롤러를 대상으로 동작하며, 컨트롤러에서 발생하는 예외를 가로채서(Intercept) 처리합니다.

  **예시**

    ```java
    @Getter
    @Builder
    public class ErrorResponse {
        private final int status;
        private final String code;
        private final String message;
    }
    ```

    ```java
    @RestControllerAdvice
    public class GlobalExceptionHandler {
    
        // 특정 예외(CustomException) 처리
        @ExceptionHandler(CustomException.class)
        public ResponseEntity<ErrorResponse> handleCustomException(CustomException e) {
            ErrorResponse response = ErrorResponse.builder()
                    .status(e.getErrorCode().getStatus())
                    .code(e.getErrorCode().getCode())
                    .message(e.getMessage())
                    .build();
            return new ResponseEntity<>(response, HttpStatus.valueOf(e.getErrorCode().getStatus()));
        }
    
        // 그 외 모든 예외 처리
        @ExceptionHandler(Exception.class)
        protected ResponseEntity<ErrorResponse> handleException(Exception e) {
            ErrorResponse response = ErrorResponse.builder()
                    .status(HttpStatus.INTERNAL_SERVER_ERROR.value())
                    .code("INTERNAL_SERVER_ERROR")
                    .message("서버 내부 에러가 발생했습니다.")
                    .build();
            return new ResponseEntity<>(response, HttpStatus.INTERNAL_SERVER_ERROR);
        }
    }
    ```

  ### 장점

    - **관심사의 분리:** 컨트롤러는 비즈니스 로직에만 집중하고, 예외 처리는 전역 어드바이스가 담당합니다.
    - **일관된 응답 포맷:** 에러가 발생했을 때 클라이언트에게 항상 일정한 형태의 JSON 응답을 내려줄 수 있습니다.
    - **범위 설정 가능:** 특정 패키지(`basePackages`)나 특정 어노테이션(`annotations`)이 붙은 컨트롤러만 대상으로 지정할 수 있습니다.

  ### 요약

    - **@ControllerAdvice vs @RestControllerAdvice:** 뷰(HTML)를 반환하는 프로젝트라면 전자를, JSON 응답을 주로 하는 REST API 프로젝트라면 후자를 사용합니다.
    - **우선순위:** 여러 개의 Advice가 있을 경우 `@Order` 어노테이션으로 우선순위를 정할 수 있지만, 가급적 **하나의 GlobalExceptionHandler**에서 관리하는 것이 흐름 파악에 유리합니다.
    - **Validation 처리:** `@Valid` 어노테이션 사용 시 발생하는 `MethodArgumentNotValidException`도 여기서 한꺼번에 처리하면 매우 편리합니다.
- Optional이란?

  `Optional<T>`는 Java 8에서 도입된 클래스로, "값이 있을 수도 있고, 없을 수도 있는 객체"를 감싸는 래퍼(Wrapper) 클래스

  비어있는 값을 다룰 때 발생할 수 있는 `NullPointerException`(NPE)을 방지하고 코드의 가독성을 높임

  | **메서드** | **설명** |
      | --- | --- |
  | **`Optional.of(value)`** | 값이 확실히 있을 때 생성 (null이면 NPE 발생) |
  | **`Optional.ofNullable(value)`** | 값이 null일 수도 있을 때 생성 (가장 많이 사용) |
  | **`isPresent()` / `isEmpty()`** | 값이 있는지/비어있는지 확인 (boolean 반환) |
  | **`ifPresent(consumer)`** | 값이 있을 때만 특정 로직 실행 |
  | **`orElse(value)`** | 값이 없으면 기본값(value)을 반환 |
  | **`orElseGet(supplier)`** | 값이 없으면 함수를 실행하여 결과값 반환 (지연 로딩) |
  | **`orElseThrow(exception)`** | 값이 없으면 예외를 발생시킴 |

  ### 예시

    ```java
    public interface MemberRepository extends JpaRepository<Member, Long> {
        // 결과가 없을 수도 있으므로 Optional로 반환
        Optional<Member> findByEmail(String email);
    }
    ```

    ```java
    @Service
    @RequiredArgsConstructor
    public class MemberService {
        private final MemberRepository memberRepository;
    
        public Member getMemberProfile(String email) {
            return memberRepository.findByEmail(email)
                    .filter(m -> m.getAge() > 20) // 조건 필터링 가능
                    .orElseThrow(() -> new CustomException("해당 이메일의 성인 회원이 없습니다."));
        }
    }
    ```

  ### 요약

    - **Optional은 "값이 없음"을 API 설계 단계에서 명시적으로 드러내는 도구**입니다.
    - **리턴 타입**으로만 사용하는 것이 권장됩니다.
    - 함수형 인터페이스(map, filter 등)와 결합하여 `if`문 없는 깔끔한 코드를 작성할 수 있게 해줍니다.
