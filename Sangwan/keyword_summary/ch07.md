- Page와 Slice
    
  ---

  > **`Page`와 `Slice`는 Spring Data JPA에서 페이지네이션(Pagination)을 구현할 때 사용하는 두 가지 인터페이스이다.**
  >
    
  ---

  ### 페이지네이션이란?

  대량의 데이터를 한 번에 조회하지 않고, **일정 단위로 나누어 조회**하는 기법이다.

  API에서 수천 건의 데이터를 한 번에 응답하면 성능 저하와 메모리 낭비가 발생하기 때문에, 페이지네이션을 통해 필요한 만큼만 가져오는 것이 일반적이다.
    
  ---

  ### Page

  `Page<T>`는 `Slice<T>`를 상속한 인터페이스로, **전체 데이터 개수(totalElements)와 전체 페이지 수(totalPages)를 함께 제공**한다.

    ```java
    // Repository
    Page<Mission> findByStore(Store store, Pageable pageable);
    
    // Service
    Pageable pageable = PageRequest.of(page, 10, Sort.by("createdAt").descending());
    Page<Mission> missionPage = missionRepository.findByStore(store, pageable);
    
    // Page가 제공하는 주요 메서드
    missionPage.getContent();       // 현재 페이지 데이터 목록
    missionPage.getTotalElements(); // 전체 데이터 수
    missionPage.getTotalPages();    // 전체 페이지 수
    missionPage.getNumber();        // 현재 페이지 번호 (0부터 시작)
    missionPage.getSize();          // 페이지 크기
    missionPage.hasNext();          // 다음 페이지 존재 여부
    missionPage.isFirst();          // 첫 번째 페이지 여부
    missionPage.isLast();           // 마지막 페이지 여부
    ```

  내부적으로 `COUNT` 쿼리를 추가로 실행하여 전체 데이터 수를 구한다.

  따라서 **오프셋 기반 페이지네이션**에 적합하다.
    
  ---

  ### Slice

  `Slice<T>`는 **다음 페이지 존재 여부만 알 수 있는** 인터페이스다.

  전체 데이터 수를 구하기 위한 `COUNT` 쿼리를 실행하지 않기 때문에 `Page`보다 성능이 좋다.

    ```java
    // Repository
    Slice<Review> findByMember(Member member, Pageable pageable);
    
    // Service
    Pageable pageable = PageRequest.of(0, 10);
    Slice<Review> reviewSlice = reviewRepository.findByMember(member, pageable);
    
    // Slice가 제공하는 주요 메서드
    reviewSlice.getContent();   // 현재 페이지 데이터 목록
    reviewSlice.hasNext();      // 다음 페이지 존재 여부
    reviewSlice.getNumber();    // 현재 페이지 번호
    reviewSlice.getSize();      // 페이지 크기
    ```

  내부적으로 `size + 1`개를 조회해서 다음 페이지 존재 여부를 판단한다.

  **커서 기반 페이지네이션**이나 무한 스크롤 방식에 적합하다.
    
  ---

  ### Page vs Slice 비교

  | 항목 | Page | Slice |
      | --- | --- | --- |
  | 전체 데이터 수 제공 | O (totalElements) | X |
  | 전체 페이지 수 제공 | O (totalPages) | X |
  | COUNT 쿼리 실행 | O (추가 쿼리 발생) | X |
  | 성능 | 상대적으로 느림 | 상대적으로 빠름 |
  | 적합한 방식 | 오프셋 기반 페이지네이션 | 커서 기반, 무한 스크롤 |
    
  ---

  ### 오프셋 기반 vs 커서 기반 페이지네이션

  **오프셋 기반**은 `LIMIT`과 `OFFSET`을 이용해 특정 위치부터 데이터를 가져오는 방식이다.

  구현이 간단하지만, 데이터가 많아질수록 `OFFSET` 값이 커져 성능이 저하된다.

  **커서 기반**은 마지막으로 조회한 데이터의 ID(커서)를 기준으로 그 다음 데이터를 가져오는 방식이다.

  `OFFSET` 없이 `WHERE id > ?` 형태로 조회하기 때문에 성능이 일정하게 유지된다.

    ```java
    // 커서 기반 예시 (Slice 활용)
    Slice<Review> findByMemberAndIdGreaterThan(Member member, Long cursor, Pageable pageable);
    ```

  > 권장 전략: 페이지 번호를 표시해야 하는 경우 `Page`, 무한 스크롤이나 "더 보기" 방식이면 `Slice`를 사용한다.

- Java stream API
    
  ---

  > **Java Stream API는 Java 8에서 도입된 기능으로, 컬렉션(Collection)이나 배열 등의 데이터를 선언형(함수형)으로 처리할 수 있게 해주는 API이다.**
  >
    
  ---

  ### Stream이란?

  Stream은 **데이터의 흐름**이다. 원본 데이터를 변경하지 않고, 파이프라인 방식으로 데이터를 처리한다.

  기존의 for문이나 Iterator 방식보다 코드가 간결하고 가독성이 높다.

    ```java
    // 기존 방식 (for문)
    List<String> result = new ArrayList<>();
    for (String name : names) {
        if (name.startsWith("김")) {
            result.add(name.toUpperCase());
        }
    }
    
    // Stream 방식
    List<String> result = names.stream()
        .filter(name -> name.startsWith("김"))
        .map(String::toUpperCase)
        .collect(Collectors.toList());
    ```
    
  ---

  ### Stream의 3단계 구조

  Stream은 **생성 → 중간 연산 → 최종 연산** 3단계로 구성된다.

  **1. 생성 (Source)**

  스트림을 생성하는 단계다.

    ```java
    List<String> list = Arrays.asList("a", "b", "c");
    Stream<String> stream = list.stream();                 // 컬렉션에서 생성
    Stream<String> parallelStream = list.parallelStream(); // 병렬 스트림
    Stream<Integer> intStream = Stream.of(1, 2, 3);        // 직접 생성
    ```

  **2. 중간 연산 (Intermediate Operation)**

  스트림을 변환하거나 필터링하는 단계다. **지연 실행(Lazy Evaluation)** 방식으로 동작하며, 최종 연산이 호출될 때까지 실행되지 않는다.

    ```java
    stream.filter(s -> s.length() > 3)   // 조건에 맞는 요소만 통과
          .map(String::toUpperCase)        // 각 요소를 변환
          .sorted()                        // 정렬
          .distinct()                      // 중복 제거
          .limit(5)                        // 최대 5개로 제한
          .skip(2);                        // 앞의 2개 건너뜀
    ```

  **3. 최종 연산 (Terminal Operation)**

  스트림을 소비하여 결과를 만들어내는 단계다. 최종 연산이 호출되면 스트림이 닫히고 재사용할 수 없다.

    ```java
    .collect(Collectors.toList())   // List로 수집
    .collect(Collectors.toSet())    // Set으로 수집
    .forEach(System.out::println)   // 각 요소에 동작 수행
    .count()                        // 요소 개수 반환
    .findFirst()                    // 첫 번째 요소 반환 (Optional)
    .anyMatch(s -> s.equals("a"))   // 조건에 맞는 요소가 하나라도 있으면 true
    .allMatch(s -> s.length() > 0)  // 모든 요소가 조건에 맞으면 true
    .reduce(0, Integer::sum)        // 요소를 누적하여 하나의 값으로 합산
    ```
    
  ---

  ### 주요 중간 연산 정리

  | 메서드 | 설명 | 예시 |
      | --- | --- | --- |
  | filter(Predicate) | 조건에 맞는 요소만 통과 | .filter(x -> x > 0) |
  | map(Function) | 각 요소를 다른 값으로 변환 | .map(String::length) |
  | flatMap(Function) | 중첩된 스트림을 평탄화 | .flatMap(Collection::stream) |
  | sorted() | 자연 순서로 정렬 | .sorted() |
  | distinct() | 중복 제거 | .distinct() |
  | limit(n) | 최대 n개로 제한 | .limit(10) |
  | skip(n) | 앞의 n개 건너뜀 | .skip(5) |
  | peek(Consumer) | 중간에 요소 확인 (디버깅용) | .peek(System.out::println) |
    
  ---

  ### Spring에서의 활용 예시

  Spring Data JPA와 함께 사용할 때 Stream API는 매우 유용하다.

    ```java
    // 미션 목록을 DTO로 변환
    List<MissionResponseDTO> dtoList = missionPage.getContent().stream()
        .map(mission -> MissionConverter.toMissionDTO(mission))
        .collect(Collectors.toList());
    
    // 특정 조건의 미션만 필터링
    List<Mission> activeMissions = missions.stream()
        .filter(m -> m.getStatus() == MissionStatus.ACTIVE)
        .collect(Collectors.toList());
    ```
    
  ---

  ### 주의사항

    - Stream은 **일회성**이다. 한 번 최종 연산을 수행하면 재사용할 수 없다.
    - 중간 연산은 **지연 실행**되므로 최종 연산이 없으면 아무것도 실행되지 않는다.
    - 병렬 스트림(`parallelStream()`)은 멀티스레드로 처리되어 성능을 높일 수 있지만, 순서가 보장되지 않고 공유 자원 접근 시 동기화 문제가 발생할 수 있다.
- 객체 그래프 탐색
    
  ---

  > **객체 그래프 탐색(Object Graph Navigation)이란, 연관관계로 연결된 객체들을 참조(reference)를 통해 연쇄적으로 접근하는 것을 말한다.**
  >
    
  ---

  ### 객체 그래프란?

  객체 지향 프로그래밍에서 객체들은 서로 참조를 통해 연결되어 있다.

  이 연결 구조를 그래프로 표현한 것이 **객체 그래프**이다.

    ```java
    // 예시: Member → Order → OrderItem → Product 순으로 탐색
    member.getOrders()                    // Member → Order 탐색
          .get(0)
          .getOrderItems()                // Order → OrderItem 탐색
          .get(0)
          .getProduct()                   // OrderItem → Product 탐색
          .getName();                     // Product의 이름 접근
    ```
    
  ---

  ### JPA에서의 객체 그래프 탐색

  JPA는 연관관계 매핑(`@OneToMany`, `@ManyToOne` 등)을 통해 객체 그래프 탐색을 지원한다.

  SQL에서는 JOIN을 통해 연관 테이블을 조회하지만, JPA에서는 **객체 참조**를 통해 자연스럽게 탐색할 수 있다.

    ```java
    // JPA 방식 (객체 참조로 탐색)
    Member member = memberRepository.findById(1L).get();
    List<Order> orders = member.getOrders(); // 객체 그래프 탐색
    ```
    
  ---

  ### 탐색 방향과 연관관계 주인

  객체 그래프 탐색은 **참조가 있는 방향으로만** 탐색이 가능하다.

    ```java
    // 단방향 매핑 (Order → Member 방향만 가능)
    public class Order {
        @ManyToOne(fetch = FetchType.LAZY)
        @JoinColumn(name = "member_id")
        private Member member; // Order에서 Member로 탐색 가능
    }
    
    // 양방향 매핑 (양쪽 모두 탐색 가능)
    public class Member {
        @OneToMany(mappedBy = "member")
        private List<Order> orders; // Member에서 Order로도 탐색 가능
    }
    ```

  양방향 매핑에서는 **연관관계의 주인(Owner)**을 지정해야 한다.

  `mappedBy`가 없는 쪽(외래 키를 가진 쪽)이 연관관계의 주인이 된다.
    
  ---

  ### JPQL에서의 객체 그래프 탐색

  JPQL에서는 `.`(점)을 이용해 객체 그래프를 탐색할 수 있다.

    ```java
    // 묵시적 조인 (권장하지 않음)
    @Query("SELECT o FROM Order o WHERE o.member.name = '마크'")
    // → o.member에서 내부적으로 JOIN이 발생하지만 코드에서 보이지 않음
    
    // 명시적 조인 (권장)
    @Query("SELECT o FROM Order o JOIN o.member m WHERE m.name = '마크'")
    // → JOIN을 명시적으로 작성하여 의도가 명확함
    ```

  묵시적 조인은 코드에서 JOIN이 보이지 않아 예상치 못한 성능 문제를 일으킬 수 있으므로, **명시적 조인을 사용하는 것이 권장**된다.
    
  ---

  ### 지연 로딩과 객체 그래프 탐색의 관계

  JPA의 기본 전략인 **지연 로딩(Lazy Loading)**에서는 연관 객체를 실제로 접근하는 시점에 쿼리가 발생한다.

  이로 인해 객체 그래프 탐색 중 N+1 문제가 발생할 수 있다.

    ```java
    // N+1 문제 발생 예시
    List<Member> members = memberRepository.findAll(); // 쿼리 1번
    for (Member member : members) {
        // 각 member.getOrders() 접근 시마다 쿼리 발생 → N번
        member.getOrders().size();
    }
    // 총 1 + N번 쿼리 발생
    ```

  이를 해결하기 위해 **패치 조인(Fetch Join)**을 사용하면 한 번의 쿼리로 연관 객체를 함께 조회할 수 있다.

    ```java
    @Query("SELECT m FROM Member m JOIN FETCH m.orders")
    List<Member> findAllWithOrders();
    ```

  ### 정리

  | 개념 | 설명 |
      | --- | --- |
  | 객체 그래프 | 객체들이 참조로 연결된 구조 |
  | 탐색 방향 | 참조가 있는 방향으로만 가능 |
  | 지연 로딩 | 실제 접근 시점에 쿼리 발생 |
  | N+1 문제 | 객체 그래프 탐색 중 발생하는 성능 문제 |
  | 패치 조인 | N+1 해결을 위한 한 번의 쿼리로 함께 조회 |
- @Valid vs @Validated
    
  ---

  > **`@Valid`와 `@Validated`는 모두 Spring에서 요청 데이터의 유효성 검사(Validation)를 수행하는 어노테이션이다. 두 어노테이션은 비슷해 보이지만 출처와 기능에서 차이가 있다.**
  >
    
  ---

  ### 유효성 검사(Validation)란?

  클라이언트로부터 들어온 요청 데이터가 **올바른 형식과 값인지 검증**하는 과정이다.

  예를 들어 이메일 형식이 맞는지, 나이가 0 이상인지 등을 검사한다.

    ```java
    // DTO에 검증 어노테이션 적용
    public class MissionRequestDTO {
        @NotNull(message = "포인트는 필수입니다.")
        @Min(value = 0, message = "포인트는 0 이상이어야 합니다.")
        private Integer point;
    
        @NotBlank(message = "조건은 필수입니다.")
        private String conditional;
    
        @Future(message = "마감일은 미래 날짜여야 합니다.")
        private LocalDate deadline;
    }
    ```
    
  ---

  ### @Valid

  `@Valid`는 **Jakarta Bean Validation(JSR-303/JSR-380)** 표준 스펙에서 제공하는 어노테이션이다.

  `jakarta.validation.Valid` 패키지에 속하며, Spring이 아닌 **Java 표준**에 정의되어 있다.

    ```java
    // Controller에서 @Valid 사용
    @PostMapping("/missions")
    public ApiResponse<MissionResponseDTO> createMission(
        @RequestBody @Valid MissionRequestDTO request) {
        // request의 필드들이 DTO에 선언된 제약 조건으로 검증됨
        return ApiResponse.onSuccess(missionService.createMission(request));
    }
    ```

    - `@RequestBody`, `@ModelAttribute` 앞에 붙여 사용한다.
    - 검증 실패 시 `MethodArgumentNotValidException`이 발생한다.
    - **중첩 객체(Nested Object) 검증**을 위해서는 해당 필드에도 `@Valid`를 붙여야 한다.

    ---

  ### @Validated

  `@Validated`는 **Spring Framework**에서 제공하는 어노테이션이다.

  `org.springframework.validation.annotation.Validated` 패키지에 속한다.

  `@Valid`의 기능을 포함하면서, **그룹 검증(Group Validation)** 기능을 추가로 지원한다.

    ```java
    // 그룹 인터페이스 정의
    public interface CreateGroup {}
    public interface UpdateGroup {}
    
    // DTO에 그룹별 제약 조건 설정
    public class MemberRequestDTO {
        @NotNull(groups = CreateGroup.class)
        private String name;
    
        @NotNull(groups = {CreateGroup.class, UpdateGroup.class})
        private String email;
    }
    
    // Controller에서 그룹 지정
    @PostMapping("/members")
    public ApiResponse<?> createMember(
        @RequestBody @Validated(CreateGroup.class) MemberRequestDTO request) {
        // CreateGroup에 해당하는 제약 조건만 검증
    }
    ```

  또한 `@Validated`는 **클래스 레벨**에 붙여 메서드 파라미터 검증도 가능하다.

    ```java
    @Service
    @Validated // 클래스 레벨에 붙이면 메서드 파라미터도 검증 가능
    public class MissionService {
        public void createMission(@NotNull Long storeId, @Valid MissionRequestDTO request) {
            // storeId가 null이면 ConstraintViolationException 발생
        }
    }
    ```
    
  ---

  ### @Valid vs @Validated 비교

  | 항목 | @Valid | @Validated |
      | --- | --- | --- |
  | 출처 | Jakarta Bean Validation (Java 표준) | Spring Framework |
  | 패키지 | jakarta.validation.Valid | org.springframework.validation.annotation.Validated |
  | 그룹 검증 | 지원 안 함 | 지원 (groups 속성 사용) |
  | 중첩 객체 검증 | 지원 (@Valid 재귀 적용) | 지원 안 함 |
  | 클래스 레벨 사용 | 불가 | 가능 (메서드 파라미터 검증) |
  | 검증 실패 예외 | MethodArgumentNotValidException | ConstraintViolationException |

  ### 주요 Bean Validation 어노테이션

  | 어노테이션 | 설명 | 예시 |
      | --- | --- | --- |
  | @NotNull | null 불허 | @NotNull private String name |
  | @NotBlank | null, 빈 문자열, 공백 불허 | @NotBlank private String email |
  | @NotEmpty | null, 빈 컬렉션/문자열 불허 | @NotEmpty private List tags |
  | @Min(value) | 최솟값 이상 | @Min(0) private Integer point |
  | @Max(value) | 최댓값 이하 | @Max(100) private Integer score |
  | @Size(min, max) | 길이 범위 | @Size(min=2, max=10) private String name |
  | @Email | 이메일 형식 | @Email private String email |
  | @Future | 미래 날짜 | @Future private LocalDate deadline |
  | @Past | 과거 날짜 | @Past private LocalDate birthDate |
  | @Positive | 양수 | @Positive private Integer count |

  ### 권장 사용 방법

    - **일반적인 요청 검증**에는 `@Valid`를 사용한다.
    - **그룹별로 다른 검증 규칙**이 필요한 경우 `@Validated`를 사용한다.
    - **Service 계층의 메서드 파라미터 검증**이 필요한 경우 클래스에 `@Validated`를 붙인다.
    - 검증 실패 시 발생하는 예외를 `@ExceptionHandler`나 `@RestControllerAdvice`로 처리하여 일관된 에러 응답을 반환하는 것이 좋다.

    ```java
    @RestControllerAdvice
    public class GlobalExceptionHandler {
        @ExceptionHandler(MethodArgumentNotValidException.class)
        public ApiResponse<?> handleValidException(MethodArgumentNotValidException e) {
            // @Valid 검증 실패 처리
        }
    
        @ExceptionHandler(ConstraintViolationException.class)
        public ApiResponse<?> handleConstraintException(ConstraintViolationException e) {
            // @Validated 검증 실패 처리
        }
    }
    ```
