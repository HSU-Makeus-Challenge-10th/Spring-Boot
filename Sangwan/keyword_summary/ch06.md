- JPA란?
    
  ---

  > **JPA**는 자바 진영에서 ORM 기술의 표준으로 사용되는 **인터페이스 명세의 모음**이다.
  >
    
  ---

  ### 등장 배경: 패러다임 불일치

  자바는 **객체 지향 언어**, DB는 **관계형 모델** → 둘의 목표가 다르기 때문에 구조적 불일치 발생

  | 구분 | 자바 | RDB |
      | --- | --- | --- |
  | 연관 관계 표현 | 참조 변수 (`A → B`) | 외래 키 하나로 양방향 가능 |
  | 상속 | 존재함 | 없음 |
  | 타입 | `LocalDateTime`, `enum` 등 | `DATE`, `NUMBER` 등 스키마 기반 |

  → 기존에는 개발자가 직접 SQL을 짜고, 자바 객체 ↔ SQL 변환을 수동으로 해야 했음
    
  ---

  ### ORM이란?

  **ORM**이란 객체와 관계형 DB 테이블을 자동으로 연결해주는 기술이다.

    - 반복적인 CRUD SQL을 직접 짜지 않아도 됨
    - 객체를 중심으로 비즈니스 로직에 집중 가능

    ---

  ### JPA의 포지션

    ```
    Java Application
          ↓
      [Spring Data JPA]  ← 가장 추상화 레벨 높음 (Spring 제공)
          ↓
       [Hibernate]       ← JPA의 구현체 (가장 널리 사용)
          ↓
         [JPA]           ← 인터페이스/표준 명세
          ↓
        [JDBC]           ← 실제 DB 통신
          ↓
         DB (MySQL 등)
    ```

  > 추상화 정도: `Spring Data JPA` > `Hibernate` > `JPA`
  >
    
  ---

  ### JPA는 라이브러리가 아니라 인터페이스

    - JPA 자체는 실제 구현이 없는 **명세**임
    - 실제로 동작하려면 **구현체**가 필요함
    - 대표 구현체: **Hibernate** (그 외 EclipseLink, DataNucleus 등)

    ```
    JPA (인터페이스)  ←→  Hibernate (구현체)
    자바의 interface  ←→  그걸 implements 한 class
    ```
    
  ---

  ### JPA의 핵심 특징 2가지

  **1. 영속성 컨텍스트**

  엔티티 객체를 관리하고 변경 내용을 추적하는 **메모리상의 논리적 공간** (= 1차 캐시 역할)

    - 한 번 조회한 엔티티는 영속성 컨텍스트에 캐싱
    - 같은 트랜잭션 내에서 동일 객체 재조회 시 DB 쿼리 없이 캐시에서 반환
    - `EntityManager`로 접근

  엔티티 생명주기

  | 상태 | 설명 |
      | --- | --- |
  | 비영속 | 영속성 컨텍스트와 무관한 새 객체 |
  | 영속 | 영속성 컨텍스트가 관리 중인 상태 |
  | 준영속 | 영속성 컨텍스트에서 분리된 상태 |
  | 삭제 | 삭제된 상태 |

  **2. ORM 자동 매핑**

    - 객체의 변경 사항을 자동 감지 → **더티 체킹**
    - 트랜잭션 커밋 시점에 모아둔 SQL 일괄 처리 → **쓰기 지연 SQL 저장소**

    ---

  ### JPA 사용 시 장점

    1. **생산성** : 반복적인 CRUD SQL 작성 불필요
    2. **유지보수** : 필드 추가 시 SQL이 아닌 클래스만 수정
    3. **DB 독립성** : 쿼리를 JPA가 처리하므로 DBMS 교체 용이
    4. **패러다임 불일치 해결** : 객체 중심 개발 가능

    ---

  ### JPA vs Hibernate vs Spring Data JPA

  |  | JPA | Hibernate | Spring Data JPA |
      | --- | --- | --- | --- |
  | 정체 | 자바 표준 인터페이스 | JPA 구현체 | Spring 제공 모듈 |
  | 제공 주체 | Jakarta EE | 오픈소스 | Spring Framework |
  | 역할 | ORM 사용 방식 정의 | 실제 ORM 동작 | Repository 추상화 |
  | 사용 방식 | `EntityManager` 직접 사용 | `Session` 사용 | `Repository` 인터페이스 메서드 사용 |

  > 실무에서는 **Spring Data JPA**를 사용하는 게 일반적
  내부적으로는 Hibernate → JPA → JDBC 순으로 동작함
  >
    
  ---

- N+1 문제란?
    
  ---

  > 연관관계가 설정된 엔티티를 조회할 때, 조회된 데이터 수(N)만큼 연관 엔티티를 조회하는 쿼리가 추가로 발생하는 성능 문제
  >

  즉, 1번의 쿼리로 끝날 것을 N번 더 날리게 되어 총 N+1번의 쿼리가 실행되는 현상이다.
    
  ---

  ### 발생 원인

  JPQL은 실행 시 즉시 로딩/지연 로딩 같은 글로벌 패치 전략을 무시하고, JPQL 자체만 분석해서 SQL을 생성한다.

  따라서 연관 엔티티는 별도로 추가 조회하게 된다.
    
  ---

  ### 예시

  `Member` 1명이 여러 개의 `Order`를 가지는 구조에서 `Member` 전체를 조회한다고 하자.

    ```
    1. SELECT * FROM member           → Member 전체 조회 (1번)
    2. SELECT * FROM orders WHERE member_id = 1  → Member1의 Order 조회
    3. SELECT * FROM orders WHERE member_id = 2  → Member2의 Order 조회
    4. SELECT * FROM orders WHERE member_id = 3  → Member3의 Order 조회
    ...
    ```

  Member가 100명이면 → 쿼리 총 101번 발생 (1 + N)
    
  ---

  ### 즉시 로딩과 지연 로딩에서의 발생 시점

  | 전략 | 발생 시점 |
      | --- | --- |
  | 즉시 로딩 | JPQL 실행 직후 연관 엔티티 조회 시 즉시 발생 |
  | 지연 로딩 | 연관 엔티티를 실제로 사용하는 시점에 발생 |

  지연 로딩이라고 해서 N+1 문제가 해결되는 것이 아니다.
  발생 시점만 뒤로 미뤄질 뿐, 연관 데이터에 접근하는 순간 동일하게 N번 쿼리가 발생한다.
    
  ---

  ### 해결 방법

  **1. 패치 조인**

  JPQL에서 `join fetch` 키워드를 사용해 연관 엔티티를 한 번의 쿼리로 함께 조회한다.

    ```java
    @Query("select m from Member m join fetch m.orders")
    List<Member> findAllWithOrders();
    ```

    - 장점 : 쿼리 1번으로 해결
    - 단점 : 페이징 API 사용 불가, 컬렉션 패치 조인은 1개까지만 가능

  **2. @EntityGraph**

  `@EntityGraph`의 `attributePaths`에 즉시 로딩으로 가져올 필드를 지정한다.
  패치 조인과 유사하게 동작하지만 어노테이션으로 선언적으로 처리 가능하다.

    ```java
    @EntityGraph(attributePaths = {"orders"})
    @Query("select m from Member m")
    List<Member> findAllWithOrders();
    ```

  **3. @BatchSize / default_batch_fetch_size**

  연관 엔티티를 조회할 때 지정한 size만큼 `WHERE IN` 절로 묶어서 조회한다.
  N번 쿼리 대신 `N / size`번 쿼리로 줄여준다.

    ```java
    // 엔티티에 직접 설정
    @BatchSize(size = 100)
    @OneToMany(mappedBy = "member")
    private List<Order> orders;
    ```

    ```yaml
    # 전역 설정 (application.yml)
    spring:
      jpa:
        properties:
          hibernate:
            default_batch_fetch_size: 100
    ```

    - 장점 : 지연 로딩을 유지하면서 성능 최적화 가능, 페이징도 사용 가능
    - 단점 : 쿼리가 완전히 1번으로 줄지는 않음

    ---

  ### 해결 방법 비교

  | 방법 | 쿼리 수 | 페이징 | 컬렉션 다중 사용 |
      | --- | --- | --- | --- |
  | 패치 조인 | 1번 | 불가 | 1개까지만 가능 |
  | @EntityGraph | 1번 | 불가 | 1개까지만 가능 |
  | @BatchSize | N/size 번 | 가능 | 가능 |

  > 권장 전략 : 기본은 지연 로딩으로 설정하고, 성능 최적화가 필요한 쿼리에만 패치 조인을 적용한다.
  페이징이 필요하거나 컬렉션이 여러 개인 경우에는 `default_batch_fetch_size`를 활용한다.
>
- 지연로딩과 즉시로딩의 차이는?
    
  ---

  > 연관된 엔티티를 **언제 조회할 것인가**에 대한 전략 차이다.
  >
    
  ---

  ### 즉시 로딩

  연관 엔티티를 **주 엔티티 조회 시점에 함께** 조회한다.

    ```java
    @ManyToOne(fetch = FetchType.EAGER)
    @JoinColumn(name = "store_id")
    private Store store;
    ```

  `MemberFood`를 조회하는 순간 `Member`, `Store`도 즉시 JOIN해서 함께 가져온다.

    ```sql
    SELECT mf.*, m.*, s.*
    FROM member_food mf
    JOIN member m ON mf.member_id = m.id
    JOIN store s ON mf.store_id = s.id
    WHERE mf.id = 1
    ```

    - 장점 : 연관 데이터를 항상 함께 사용하는 경우 추가 쿼리 없이 한 번에 가져올 수 있음
    - 단점 : 연관 데이터가 필요 없는 경우에도 무조건 조회 → 불필요한 쿼리 발생
    - 단점 : JPQL 사용 시 N+1 문제 발생 가능성이 높음

    ---

  ### 지연 로딩

  연관 엔티티를 **실제로 사용하는 시점**에 조회한다.
  사용 전까지는 프록시 객체로 대체해둔다.

    ```java
    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "store_id")
    private Store store;
    ```

  `MemberFood`를 조회할 때는 `Store`를 가져오지 않는다.
  `memberFood.getStore().getName()` 처럼 실제로 접근하는 순간 추가 쿼리가 발생한다.

    ```sql
    -- MemberFood 조회 시
    SELECT * FROM member_food WHERE id = 1
    
    -- store 필드에 실제 접근하는 순간
    SELECT * FROM store WHERE id = ?
    ```

    - 장점 : 필요한 시점에만 쿼리가 나가므로 불필요한 조회를 막을 수 있음
    - 단점 : 연관 데이터에 접근할 때마다 추가 쿼리 발생 → 잘못 쓰면 N+1 문제로 이어짐

    ---

  ### 프록시란?

  지연 로딩에서 실제 객체 대신 임시로 채워두는 **껍데기 객체**다.
  실제 데이터가 필요한 시점까지 DB 조회를 미루는 역할을 한다.

    ```
    memberFood.getStore()       → 프록시 객체 반환 (DB 조회 안 함)
    memberFood.getStore().getName()  → 이 시점에 DB 조회 발생
    ```
    
  ---

  ### 기본값 정리

  | 어노테이션 | 기본 전략 |
      | --- | --- |
  | `@ManyToOne` | 즉시 로딩 |
  | `@OneToOne` | 즉시 로딩 |
  | `@OneToMany` | 지연 로딩 |
  | `@ManyToMany` | 지연 로딩 |

  `@ManyToOne`, `@OneToOne`은 기본이 즉시 로딩이라 명시적으로 `LAZY`를 설정해주지 않으면 N+1 문제에 노출된다.
    
  ---

  ### 비교 요약

  |  | 즉시 로딩 | 지연 로딩 |
      | --- | --- | --- |
  | 조회 시점 | 주 엔티티 조회 시 함께 | 연관 데이터 실제 사용 시 |
  | 초기 쿼리 수 | 많음 (JOIN 포함) | 적음 |
  | N+1 위험 | 높음 | 낮음 (하지만 존재) |
  | 적합한 상황 | 연관 데이터를 항상 같이 쓰는 경우 | 연관 데이터를 선택적으로 쓰는 경우 |
    
  ---

  ### 권장 전략

  기본은 모두 **지연 로딩**으로 설정하고, 성능 최적화가 필요한 곳에서만 **패치 조인**으로 한 번에 가져오는 방식이 권장된다.

  즉시 로딩은 예상치 못한 쿼리가 발생하기 쉽고, 연관관계가 복잡할수록 성능 문제를 추적하기 어려워진다.

- JPQL란?
    
  ---

  > **JPQL**은 JPA에서 사용하는 객체 지향 쿼리 언어로, 테이블이 아닌 **엔티티 객체를 대상**으로 쿼리를 작성한다.
  >
    
  ---

  ### SQL과의 차이

  SQL은 DB 테이블을 대상으로 쿼리를 작성하지만, JPQL은 엔티티 클래스와 필드를 대상으로 작성한다.
  JPA가 내부적으로 JPQL을 분석해서 각 DB에 맞는 SQL로 변환해 실행한다.

    ```sql
    -- SQL : 테이블과 컬럼명 기준
    SELECT * FROM member WHERE name = '마크'
    
    -- JPQL : 엔티티 클래스와 필드명 기준
    SELECT m FROM Member m WHERE m.name = '마크'
    ```

  `Member`는 테이블명이 아니라 **엔티티 클래스명**이고, `m.name`은 **엔티티 필드명**이다.
    
  ---

  ### 특징

    - DB 종류에 독립적이다. JPQL을 작성하면 JPA가 MySQL, Oracle 등 각 DB 방언에 맞게 SQL로 변환해준다.
    - 즉시 로딩/지연 로딩 같은 글로벌 패치 전략을 무시하고 항상 DB에 직접 쿼리를 날린다.
    - 영속성 컨텍스트를 거치지 않고 DB를 먼저 조회한 뒤 결과를 영속성 컨텍스트에 저장한다.

    ---

  ### Spring Data JPA에서 JPQL을 사용하는 방법

  **1. 메서드 이름으로 쿼리 자동 생성**

  Spring Data JPA가 메서드 이름을 분석해서 자동으로 JPQL을 생성해준다.

    ```java
    // name 필드와 deletedAt이 null인 조건으로 자동 생성
    List<Member> findByNameAndDeletedAtIsNull(String name);
    ```

  내부적으로 아래 쿼리가 실행된다.

    ```sql
    SELECT * FROM member WHERE name = ? AND deleted_at IS NULL
    ```

  자주 쓰이는 키워드

  | 키워드 | 설명 | 예시 |
      | --- | --- | --- |
  | `findBy` | 조회 | `findByName` |
  | `And` / `Or` | 조건 연결 | `findByNameAndEmail` |
  | `IsNull` / `IsNotNull` | null 여부 | `findByDeletedAtIsNull` |
  | `Like` | 부분 일치 | `findByNameLike` |
  | `GreaterThan` | 초과 | `findByAgeGreaterThan` |
  | `OrderBy` | 정렬 | `findByNameOrderByCreatedAtDesc` |

  **2. @Query 어노테이션으로 직접 작성**

  복잡한 조건이나 성능 최적화가 필요한 경우 JPQL을 직접 작성한다.

    ```java
    @Query("SELECT m FROM Member m WHERE m.name = :name AND m.deletedAt IS NULL")
    List<Member> findActiveByName(@Param("name") String name);
    ```

  **3. 네이티브 쿼리**

  `nativeQuery = true`로 설정하면 JPQL 대신 순수 SQL을 사용할 수 있다.

    ```java
    @Query(value = "SELECT * FROM member WHERE name = :name AND deleted_at IS NULL",
           nativeQuery = true)
    List<Member> findActiveByNameNative(@Param("name") String name);
    ```
    
  ---

  ### 패치 조인

  JPQL에서 연관 엔티티를 한 번의 쿼리로 함께 조회하는 문법이다.
  N+1 문제를 해결할 때 가장 많이 사용된다.

    ```java
    @Query("SELECT m FROM Member m JOIN FETCH m.orders")
    List<Member> findAllWithOrders();
    ```

  일반 `JOIN`은 연관 엔티티를 조회 조건으로만 사용하지만, `JOIN FETCH`는 연관 엔티티까지 실제로 함께 가져온다.
    
  ---

  ### 주의사항

    - 엔티티 클래스명과 필드명은 **대소문자를 구분**한다.
    - 테이블명, 컬럼명이 아니라 반드시 **엔티티명, 필드명**을 사용해야 한다.
    - JPQL은 항상 DB를 직접 조회하므로 영속성 컨텍스트에 이미 같은 데이터가 있어도 쿼리가 나간다. 조회 후 영속성 컨텍스트에 이미 해당 엔티티가 존재하면 DB에서 가져온 값은 버리고 기존 캐시 값을 유지한다.
- Fetch Join란?
    
  ---

  > JPQL에서 연관된 엔티티를 **한 번의 쿼리로 함께 조회**하기 위한 문법이다.
  SQL의 JOIN과 달리, 연관 엔티티를 실제로 영속성 컨텍스트에 올려준다.
  >
    
  ---

  ### 일반 JOIN과의 차이

    ```sql
    -- 일반 JOIN : 조회 조건에만 사용, 연관 엔티티는 가져오지 않음
    SELECT m FROM Member m JOIN m.orders o WHERE o.status = 'DONE'
    
    -- 패치 조인 : 연관 엔티티까지 실제로 함께 조회
    SELECT m FROM Member m JOIN FETCH m.orders
    ```

  일반 JOIN은 WHERE 조건이나 정렬 등에 연관 엔티티를 활용할 뿐, 실제 데이터를 가져오지는 않는다.
  패치 조인은 연관 엔티티까지 한 번에 SELECT해서 영속성 컨텍스트에 올린다.
    
  ---

  ### 사용 방법

    ```java
    @Query("SELECT m FROM Member m JOIN FETCH m.orders")
    List<Member> findAllWithOrders();
    ```

  실행되는 SQL

    ```sql
    SELECT m.*, o.*
    FROM member m
    INNER JOIN orders o ON o.member_id = m.id
    ```

  Member를 조회하면서 연관된 Order도 한 번에 가져오므로 N+1 문제가 발생하지 않는다.
    
  ---

  ### 컬렉션 패치 조인과 중복 문제

  `@OneToMany` 관계에서 패치 조인을 사용하면 조인 결과로 인해 **중복 데이터**가 발생할 수 있다.

  예를 들어 Member 1명이 Order 3개를 가지면 조인 결과는 3개의 행이 된다.
  JPA는 이를 그대로 받아 Member 엔티티를 3개 반환한다.

  이를 방지하려면 JPQL에 `DISTINCT`를 추가한다.

    ```java
    @Query("SELECT DISTINCT m FROM Member m JOIN FETCH m.orders")
    List<Member> findAllWithOrders();
    ```

  SQL의 DISTINCT와 다르게, JPQL의 DISTINCT는 **애플리케이션 레벨에서도 중복 엔티티를 제거**해준다.
    
  ---

  ### 한계와 주의사항

  **1. 페이징 불가**

  컬렉션 패치 조인에서는 페이징 API를 사용할 수 없다.

  JPA가 조인 결과 전체를 메모리에 올린 뒤 애플리케이션에서 페이징을 처리하기 때문에
  데이터가 많으면 메모리 부족 문제가 발생할 수 있다.

  단일 값 연관 필드(`@ManyToOne`, `@OneToOne`)는 페이징이 가능하다.

  **2. 컬렉션 패치 조인은 1개까지만 가능**

  `@OneToMany`나 `@ManyToMany` 같은 컬렉션 패치 조인은 한 쿼리에서 **1개만** 사용할 수 있다.
  2개 이상 사용하면 데이터 정합성 문제가 발생한다.

  **3. 지연 로딩 설정이 무의미해짐**

  패치 조인을 사용하면 `FetchType.LAZY`로 설정해두어도 해당 쿼리에서는 즉시 함께 조회한다.
    
  ---

  ### 페이징이 필요한 경우 대안

  컬렉션 패치 조인에서 페이징까지 필요하다면 `default_batch_fetch_size`를 사용한다.

    ```yaml
    spring:
      jpa:
        properties:
          hibernate:
            default_batch_fetch_size: 100
    ```

  지연 로딩을 유지하면서 연관 엔티티를 `WHERE IN` 절로 묶어서 조회하므로
  페이징도 가능하고 쿼리 수도 크게 줄일 수 있다.
    
  ---

  ### 정리

  |  | 일반 JOIN | 패치 조인 |
      | --- | --- | --- |
  | 연관 엔티티 조회 여부 | 조회 안 함 | 함께 조회 |
  | N+1 해결 | 안 됨 | 됨 |
  | 페이징 | 가능 | 컬렉션에서 불가 |
  | 컬렉션 다중 사용 | 가능 | 1개까지만 가능 |

  > 기본 전략은 지연 로딩으로 설정하고, 성능 최적화가 필요한 쿼리에만 패치 조인을 적용하는 것이 권장된다.
>
- @EntityGraph란?
    
  ---

  > 엔티티 조회 시 연관 엔티티를 **어떻게 로딩할지 선언적으로 지정**하는 어노테이션이다.
  패치 조인과 동일하게 한 번의 쿼리로 연관 엔티티를 함께 조회하지만, JPQL을 직접 작성하지 않아도 된다.
  >
    
  ---

  ### 패치 조인과의 차이

  패치 조인은 `@Query`에 JPQL을 직접 작성해야 한다.
  `@EntityGraph`는 어노테이션만 붙이면 Spring Data JPA가 자동으로 처리해준다.

    ```java
    // 패치 조인 방식
    @Query("SELECT m FROM Member m JOIN FETCH m.orders")
    List<Member> findAllWithOrders();
    
    // @EntityGraph 방식
    @EntityGraph(attributePaths = {"orders"})
    @Query("SELECT m FROM Member m")
    List<Member> findAllWithOrders();
    
    // 메서드 이름 방식과도 함께 쓸 수 있음
    @EntityGraph(attributePaths = {"orders"})
    List<Member> findAll();
    ```
    
  ---

  ### 사용 방법

  `attributePaths`에 즉시 로딩으로 가져올 필드명을 지정한다.

    ```java
    public interface MemberRepository extends JpaRepository<Member, Long> {
    
        @EntityGraph(attributePaths = {"orders"})
        List<Member> findAll();
    }
    ```

  실행되는 SQL

    ```sql
    SELECT m.*, o.*
    FROM member m
    LEFT OUTER JOIN orders o ON o.member_id = m.id
    ```

  패치 조인은 `INNER JOIN`을 사용하지만, `@EntityGraph`는 기본적으로 **`LEFT OUTER JOIN`** 을 사용한다.
    
  ---

  ### 중첩된 연관관계 처리

  연관관계의 연관관계까지 함께 가져와야 할 때는 `.`으로 경로를 이어서 작성한다.

    ```java
    // Member → orders → store 까지 함께 조회
    @EntityGraph(attributePaths = {"orders", "orders.store"})
    List<Member> findAll();
    ```
    
  ---

  ### 한계와 주의사항

  **1. 컬렉션에서 페이징 불가**

  패치 조인과 동일하게, `@OneToMany` 컬렉션에 `@EntityGraph`를 사용하면 페이징이 불가능하다.
  메모리에서 전체 데이터를 로딩한 뒤 페이징을 처리하므로 데이터가 많으면 위험하다.

  **2. 중복 데이터 발생 가능**

  컬렉션 조회 시 JOIN 결과로 중복 엔티티가 발생할 수 있다.
  `DISTINCT`를 JPQL에 추가하거나, `Set`으로 받아서 처리한다.

    ```java
    @EntityGraph(attributePaths = {"orders"})
    @Query("SELECT DISTINCT m FROM Member m")
    List<Member> findAll();
    ```

  **3. 컬렉션은 1개까지만 가능**

  패치 조인과 동일하게 컬렉션 타입의 연관관계는 한 번에 1개까지만 지정할 수 있다.
    
  ---

  ### 패치 조인 vs @EntityGraph 비교

  |  | 패치 조인 | @EntityGraph |
      | --- | --- | --- |
  | 작성 방식 | JPQL 직접 작성 | 어노테이션 선언 |
  | JOIN 방식 | INNER JOIN | LEFT OUTER JOIN |
  | 메서드 이름 쿼리와 병행 | 불가 | 가능 |
  | 복잡한 쿼리 처리 | 유리 | 단순한 경우 유리 |
  | 페이징 | 컬렉션에서 불가 | 컬렉션에서 불가 |
    
  ---

  ### 언제 쓸까

    - 단순한 연관관계 조회에서 JPQL 없이 N+1을 해결하고 싶을 때
    - 기존 메서드 이름 쿼리(`findAll`, `findById` 등)에 연관 엔티티 로딩을 추가하고 싶을 때
    - 복잡한 조건이 필요하다면 `@Query` + 패치 조인이 더 적합하다
- commit과 flush 차이점은?
    
  ---

  > 둘 다 영속성 컨텍스트의 변경 내용을 DB에 반영하는 과정에서 등장하지만, 역할과 범위가 다르다.
  >
    
  ---

  ### flush

  > 영속성 컨텍스트에 쌓인 변경 내용을 **즉시 DB에 SQL로 전송**하는 동작이다.
  >
    - 쓰기 지연 SQL 저장소에 모아둔 INSERT, UPDATE, DELETE 쿼리를 DB로 날린다.
    - **트랜잭션은 아직 끝나지 않는다.** DB에 쿼리가 전달되지만 아직 확정된 상태가 아니다.
    - 영속성 컨텍스트의 1차 캐시는 그대로 유지된다.

    ```java
    em.flush(); // 쿼리는 DB로 전송되지만 트랜잭션은 아직 열려있음
    ```

  flush가 발생하는 시점

    - `em.flush()` 직접 호출
    - 트랜잭션 커밋 시 내부적으로 자동 호출
    - JPQL 실행 전 자동 호출 (데이터 정합성 보장을 위해)

    ---

  ### commit

  > 트랜잭션을 **최종 확정**하는 동작이다.
  >
    - 내부적으로 flush를 먼저 실행해서 변경 내용을 DB로 전송한다.
    - 그 다음 트랜잭션을 확정(commit)해서 DB에 영구 반영한다.
    - commit이 완료되면 다른 트랜잭션에서도 변경 내용이 보이게 된다.

    ```java
    em.getTransaction().commit(); // flush → commit 순으로 동작
    ```
    
  ---

  ### 동작 순서

    ```
    1. 엔티티 변경
          ↓
    2. 쓰기 지연 SQL 저장소에 쿼리 적재
          ↓
    3. flush() → DB에 SQL 전송 (아직 트랜잭션 열려있음)
          ↓
    4. commit() → 트랜잭션 확정, DB에 영구 반영
    ```

  commit을 호출하면 3번과 4번이 순서대로 자동 실행된다.
    
  ---

  ### rollback과의 관계

  flush 후 commit 전에 예외가 발생하면 rollback이 가능하다.
  flush는 SQL을 DB에 보낸 것이지 확정한 것이 아니기 때문이다.

    ```java
    try {
        em.getTransaction().begin();
    
        member.setName("변경된이름"); // 영속성 컨텍스트에 변경 감지
        em.flush();                  // DB에 UPDATE 쿼리 전송 (아직 확정 아님)
    
        em.getTransaction().commit(); // 트랜잭션 확정
    } catch (Exception e) {
        em.getTransaction().rollback(); // flush 이후라도 rollback 가능
    }
    ```
    
  ---

  ### 비교 요약

  |  | flush | commit |
      | --- | --- | --- |
  | 역할 | 쌓인 SQL을 DB로 전송 | 트랜잭션 최종 확정 |
  | 트랜잭션 종료 여부 | 종료 안 됨 | 종료됨 |
  | 1차 캐시 유지 여부 | 유지됨 | 종료됨 |
  | rollback 가능 여부 | 가능 | 불가 |
  | 자동 호출 시점 | commit 직전, JPQL 실행 전 | 직접 호출 또는 트랜잭션 종료 시 |
