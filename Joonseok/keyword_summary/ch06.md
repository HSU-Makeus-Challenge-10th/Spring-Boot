- JPA란?

    <aside>
    ✨

  #### SQL을 모두 적던 시대여 안녕…

    </aside>

  JPA(Java Persistence API)는 객체 지향 패러다임과 관계형 데이터베이스의 철학 충돌로 인해 발생한 개발 비용 증가 문제를 해결하기 위하여 개발된 객체 - 관계 매핑 기술 중, 현재 Java 진영에서 채택한 기술입니다.

  표준 구현체로 Hibernate를 사용하며, 클래스로 테이블을 관리하고, 함수로 SQL을 호출할 수 있게 합니다. 이 덕분에 개발자는 테이블의 사소한 수정에도 모든 SQL을 수정하는 수고를 줄이게 되며, SQL 방언에 대해서도 구애받지 않고 개발할 수 있게 되었습니다.

    - 간단한 코드 예시…

        ```java
        @Entity
        @Getter
        @Builder
        @AllArgsConstructor(access = AccessLevel.PRIVATE)
        @NoArgsConstructor(access = AccessLevel.PROTECTED)
        public class Member {
        
        	@Id @GeneratedValue(strategy = GenerationType.IDENTITY)
        	private Long id;
        	
        	@Column(nullable = false, unique = true)
        	private String loginId  // DB field = login_id (snake case)
        	
        	// ...
        }
        ```

        ```java
        public JpaTest {
        	
        	@Autowired
        	private final EntityManagerFactory emf;
        	
        	private EntityManager = emf.createEntityManager();
        	private EntityTranscation tx = em.getTransaction();
        	
        	@Test
        	public memberSave() {
        		try {
        			// begin Transaction
        			tx.begin();
        			
        			// New member generate
        			Member member = new Member(); // 파라미터 생략
        			
        			// DB save reqeuest
        			em.persist(member)
        			
        		} catch(Exception e) {
        			// if throw exception in try block, Transaction rollback
        			tx.rollback();
        			
        		} finally {
        			// complete, transaction commit
        			tx.commit();
        			
        		}
        	}
        }
        ```


    ### JPA 장점
    
    #### First-level Cache
    
    DB에 쿼리 작업을 요청하는 것은 메모리 단위에서 작업하는 것보다 큰 네트워킹 작업이므로, 운영체제에서 사용자 라이브러리가 버퍼를 두어 시스템 호출 라이브러리보다 더 빠르게 작동하는 것과 같은 원리이다.
    
    - 동일 트랜잭션 내에서만 작동
    - Managed 상태의 객체에만 해당
    - find 함수를 호출할 때, 1차 캐시를 우선적으로 사용
        - cache hit : DB 접근 없이, 1차 캐시의 레코드 반환
        - cache miss : 이 때, DB에 접근하여 쿼리 작업
    
    #### 인스턴스 동일성 보장
    
    ```java
    public class Main {
    	
    	// 생략
    	
    	public static void main(String args[]) {
    		// transaction started...
    		Member member1 = em.find(1L, Member.class); // DB Query
    		Member member2 = em.find(1L, Member.class); // find in First-level Cache
    		
    		System.out.println(member1 == member2 ? "true" : "false");
    		// transaction committed...
    	}
    }		
    ```
    
    보통의 Java 객체는 new 키워드마다 동적으로 할당되어 레퍼런스 변수는 해당 값의 메모리 주소를 기억합니다. 하지만, 1차 캐시 덕분에 동일한 트랜잭션 안에서는 `.equals()` 비교가 아닌, `==` 비교를 사용하여 객체를 비교하여도 동일성을 보장받을 수 있습니다. JPA를 사용하여 트랜잭션 격리 수준을 `Repeatable Read` 로 설정한 것입니다. 이 역시, `Managed` 상태일 때에만 해당됩니다.
    
    #### 트랜잭션을 지원하는 쓰기 지연
    
    ```java
    public JpaTest {
    	
    	@Autowired
    	private final EntityManagerFactory emf;
    	
    	private EntityManager = emf.createEntityManager();
    	private EntityTranscation tx = em.getTransaction();
    	
    	@Test
    	public memberSave() {
    		try {
    			// begin Transaction
    			tx.begin();
    			
    			// New member generate
    			Member member1 = new Member(); // 파라미터 생략
    			Member member2 = new Member(); // 파라미터 생략
    			
    			// DB doesn't know them yet.
    			// if find() proceed, JPA returned records in 1st cache
    			
    			// DB save reqeuest
    			em.persist(member)
    			
    		} catch(Exception e) {
    			// if throw exception in try block, Transaction rollback
    			tx.rollback();
    			
    		} finally {
    			// complete, transaction commit
    			tx.commit();
    			// if commit, JPA flush lazy write repository, translate SQL, query DB
    			
    		}
    	}
    }
    ```
    
    DB 접근은 네트워킹 작업으로, 메모리 단위 작업에 비해 오버헤드가 매우 큰데, 한 트랜잭션 내에서 `persist()` 를 빈번하게 호출하게 되면, 요청을 처리하는데 속도가 현저히 느려지게 된다. 따라서, JPA는 `persist()` 의 매개로 전달된 변수를 1차캐시에 저장하고, 트랜잭션이 종료될 때까지 DB에 해당 내용을 전파하지 않는다. 1차 캐시에 해당 레코드를 저장하고, 조회 작업이 발생하면 1차 캐시의 레코드를 반환하게 되며, `rollback()`되어도 DB 작업이 아니라 메모리 단위 작업이므로 성능이 향상된다.
    
    #### Dirty Check(변경 감지)
    
    ```java
    public JpaTest {
    	
    	@Autowired
    	private final EntityManagerFactory emf;
    	
    	private EntityManager = emf.createEntityManager();
    	private EntityTranscation tx = em.getTransaction();
    	
    	@Test
    	public memberSave() {
    		try {
    			// begin Transaction
    			tx.begin();
    			
    			// Member find
    			Member member = em.find(1L, Member.class);
    			
    
    			// =======================
    			// ===== Setter call =====
    			// =======================
    			member.setName("Aden");
    
    			// shouldn't call update. because JPA!
    			
    		} catch(Exception e) {
    			// if throw exception in try block, Transaction rollback
    			tx.rollback();
    			
    		} finally {
    			// complete, transaction commit
    			tx.commit();
    			
    		}
    	}
    }
    ```
    
    JPA는 조회 시점(`find()` )의 레코드를 스냅샷으로 만들어 저장해둔다. 또, 트랜잭션이 종료될 때, 종료 시점의 레코드 값과 스냅샷을 비교하여, 변경된 점이 있는지 체크한다. 변경된 점이 있다면, 해당 값을 업데이트 하는 `UPDATE - VALUES` SQL을 작성하여, 쓰기 지연 저장소에 저장해두고, 특정 시점에 DB 작업이 스케줄링되면 그 시점에 DB에 내용을 저장하게 된다. 이를 단계로 작성하면 아래와 같다
    
    1. 이전 트랜잭션의 commit, 현재 트랜잭션 begin
    2. 레코드 조회 메소드 호출 (`em.find(nL, Entity.class`)
        1. 스냅샷 생성
    3. 레코드 수정(`entity.setField(”???”)`)
    4. 개발자가 명시적으로 update 코드를 작성 X, 그저 `tx.commit()` 호출
    5. JPA는 스냅샷과 커밋 시점의 객체를 비교
        1. 변경된 점이 있다면, 해당 내용의 `UPDATE - VALUES` SQL을 작성
    6. 쓰기 지연 저장소에 해당 SQL 저장
    7. DB 작업이 스케줄링되면, SQL을 DB에 전송하여 쿼리 진행

  ### Lazy Loading

  다른 개체와 관계가 있는 개체를 클래스로 작성할 때, 조회 옵션을 설정할 수 있다.

    - 이 엔터티를 조회할 때, 관계된 그 객체도 함께 한번에 조회할 것인지
    - 이 엔터티를 조회할 때, 관계된 그 객체는 프록시(가짜 객체)로 둘 것인지

  관계가 단순한 경우, 혹은 종속 관계가 너무 강하게 설정되어 있어, 반드시 같이 호출되어야 하는 객체가 아닌 경우, 보통 권장하는 옵션은 `fetch = FetchType.LAZY` 가 권장된다. 이 지연 로딩의 경우, 해당 필드를 프록시로 두어, 조회되기 전까지는 실제로 조회하지 않는다는 점이 특징이다.

  문제는 프록시 객체를 조회하려고 할 때이다.

    - 예시…

        ```java
        public JpaTest {
        	
        	@Autowired
        	private final EntityManagerFactory emf;
        	
        	private EntityManager = emf.createEntityManager();
        	private EntityTranscation tx = em.getTransaction();
        	
        	@Test
        	public memberSave() {
        		try {
        			// begin Transaction
        			tx.begin();
        			
        			// Member find
        			Member member = em.find(1L, Member.class);
        			
        			// first time
        			member.getTeam();
        			
        		} catch(Exception e) {
        			// if throw exception in try block, Transaction rollback
        			tx.rollback();
        			
        		} finally {
        			// complete, transaction commit
        			tx.commit();
        			
        		}
        		// second time
        		member.getTeam();
        	}
        }
        ```


    프록시 객체는 아예 SQL을 만들어 조회하지 않았기 때문에, 조회 시점에 또 새로이 SQL을 만들어서 조회 작업을 수행하게 된다.
    
    ```java
    List<Member> members = memberRepository.findAll();
    members.stream()
    	.filter(m -> { if(m.getTeam() != A) continue; return; }
    	.asList();
    }
    ```
    
    만약, Team 객체가 `fetch = FetchType.LAZY` 라고 선언되어 있다면, 해당 Stream이 실행되는 동안 Team을 조회하기 위해 멤버의 횟수만큼 더 조회하게 될 것이다. 개발자는 분명 모든 멤버를 조회하는 1개의 쿼리를 작성했을 뿐인데, 멤버의 수인 n개 만큼의 조회 작업이 추가적으로 발생하게 되는 것이다. 이 것이 바로 `N+1 쿼리 문제` 이다.

- 지연로딩과 즉시로딩의 차이는?


    | FetchType | EAGER | LAZY |
    | --- | --- | --- |
    | default | XX To One의 관계일 때 기본값 | XX To Many의 기본값 |
    | 권장 여부 | 권장되지 않음 | 권장됨 |
    | 필드 조회 | 관계의 주인 객체가 조회될 때, 같이 조회됨 | 조회 함수가 반환한 그 값에서는 프록시. 해당 필드를 조회하려 할 때 실제로 쿼리됨 |
- JPQL란?

    <aside>
    😎

  #### 나는 SQL과는 달라…

    </aside>

  JPQL(Java Persistence Query Language)은 SQL과 객체 지향 언어를 결합한 형태입니다.

    - JPA가 지원하는 객체지향 쿼리 언어
    - `객체` 를 대상으로 쿼리

  저희가 아는 SQL은 아래와 같은 형태였습니다.

    ```sql
    SELECT *
    	FROM Member AS m
    		JOIN Team AS t
    			ON (m.team_id = t.team_id)
    	WHERE m.id < 5;
    ```

  JPQL은 아래와 같이 작성합니다.

    ```sql
    SELECT m
    	FROM Member m
    		JOIN FETCH m.team
    	WHERE m.id < :member_id
    ```

  특징은 객체 그래프 탐색 형태로 JOIN 절을 작성한다는 점입니다. SQL은 ON절까지 사용하며 필드를 비교하여 연결하는 형태로 SQL을 작성했다면, JPQL은 Member 클래스 조회에 .getTeam()을 끼운 것처럼 QL을 작성했습니다.

  또, JPQL은 호출 시, 영속 컨텍스트를 우회하여 직접 DB에 쿼리를 진행합니다. 진행 단계는 아래와 같습니다.

  #### 조회의 경우

    - JPQL 쿼리 실행 함수 호출
    - 기존 영속 컨텍스트 flush
    - DB 조회 작업 진행
    - 결과를 영속 컨텍스트에 반영

  #### 생성, 수정, 삭제의 경우

    - JPQL query exec function called
    - flush
    - DB query
    - `영속 컨텍스트에 반영하지 않음!!!`
        - 1차 캐시와 해당 반환값이 일치하지 않으므로, 영속 컨텍스트를 비우는 작업 필요
- Fetch Join란?

    <aside>
    ⚠️

  ### JPA가 제공하는 성능 향상 기능!

    </aside>

  우선, fetch join은 통상적인 SQL에서 제공되는 기능이 아닙니다. JPQL에서 성능 최적화를 위해 제공되는 JOIN 키워드입니다. 연관된 객체, 컬렉션을 한 번에 조회할 수 있는 기능입니다.

  일반적인 JOIN 키워드는 연관된 객체, 컬렉션을 조회하지 않습니다. 똑같이 프록시 객체로 주입된다는 것입니다. 따라서, 프록시 객체에 접근하게 되면 마찬가지로 N+1 쿼리 문제가 발생하게 됩니다. 하지만, fetch join을 사용하게 되면 1번의 쿼리로 연관된 객체가 1개건, N개건 DB에 접근하는 행위가 1회 발생하게 된다는 것입니다.

  다만, fetch join으로 컬렉션을 조회했을 때 실행되는 결과는 다음과 같습니다.

  ### 한계

    - 기본적으로는 fetch join의 대상 객체는 `Alias` 를 부여할 수 없습니다. 잘못된 별칭 사용으로 데이터 무결성이 깨질 수 있기 때문입니다. Spring의 JPA 구현체인 Hibernate는 Alias를 부여할 수 있으나, 주의하여 사용하여야 합니다.
    - 둘 이상의 컬렉션을 fetch할 수 없습니다. 컬렉션 간의 카티션 곱이 생성되어 데이터의 양이 방대해지게 됩니다. 가능한 JPA를 사용한다고 하더라도, 주의하여 사용하여야 합니다.
    - XX To Many 필드를 Fetch join 한 경우에는 Paging API를 사용할 수 없게 됩니다. Hibernate의 경우, 경고 로그를 남기고 모든 데이터를 불러와 메모리에서 페이징을 진행하기 때문에 매우 위험한 작업입니다.
- @EntityGraph란?

    <aside>
    ⚡

  #### N+1 쿼리 문제를 해결하는 간편한 방법!!!

    </aside>

  Spring boot 기준 2.1.0, Spring Data JPA 기준 1.6 부터 도입된 기능으로, fetch 타입이 LAZY인 객체를 한번에 조회하게 해주는 어노테이션이다.

  JPQL을 작성하지 않아도, 연관 객체를 fetch join으로 조회할 수 있으므로 편리한 기능입니다.

    ```java
    @Repository // Spring Data JPA 내의 JpaRepository를 상속한 경우, 생략 가능함
    public interface MemberRepository extends JpaRepository<Member, Long> {
    	
    	@EntityGraph(attributes = {"Team"})
    	Optional<Member> findById(Long Id);
    }
    ```

- commit과 flush 차이점은?


    | flush |
    | --- |
    | 영속성 컨텍스트를 비우지 X |
    | 영속성 컨텍스트의 변경 내용을 DB 동기화 |
    | commit 호출 시, flush가 되고 난 후 트랜잭션을 commit(안전 종료)함 |
    
    Commit은 이 flush 작업이 끝난 후, DB에 해당 트랜잭션이 안전하게 종료되었음을 전파함.
    
    따라서, flush가 전부 완료되었더라도, Commit되기 전에 Rollback을 수행하면, 트랜잭션을 모두 원상태로 복귀시킨다.
