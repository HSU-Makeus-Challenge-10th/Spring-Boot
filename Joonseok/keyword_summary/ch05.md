- 빌더패턴이란?

  ### Flexible Constructor

    </aside>

  기존의 생성자 작성 방식은…

    ```java
    public class Human {
    
    	private String name;
    	private int age;
    	private double height;
    	private double weight;
    	
    	public Human(String name, int age, double height, double weight) {
    		this.name = name;
    		this.age = age;
    		this.height = height;
    		this.weight = weight;
    	}
    	
    	public Human(String name, int age) {
    		this.name = name;
    		this.age;
    		this.height = null;
    		this.weight = null;
    	}
    	
    	// ...생략...
    }
    ```

  **`생성자 오버로딩`** 방식으로 여러 상황에 대응할 수 있도록 했습니다.

  ### 단!

  이러한 생성자 선언 방식은, 매개변수의 순서에 따라, 받아들이는 매개변수의 개수 등 여러 요인에 의해 강제되며, 모든 상황에 대응하려고 한다면 무의미하게 코드가 늘어나는 문제가 발생할 수 있습니다.

  ### 빌더 패턴의 유용성

  빌더 패턴은, 매개변수의 순서에 구애받지 않으며, 특정 필드를 의도적으로 누락하여도 객체를 빌드할 수 있는 패턴입니다.

    ```java
    public class Main {
    	public static void main(String args[]) {
    	
    		// 기존의 Human 객체 생성법
    		Human instance = new Human("John doe", 20, 180.0, 72.0);
    		
    		// Builder pattern 적용
    		Human human = Human.builder()
    			.name("John doe")
    			// 선언하지 않은 생성자 매개변수 순서대로 객체 생성 가능!
    			.height(180.0)
    			.weight(72.0)
    			.build()
    	
    	}
    }
    ```

  그렇지만 단점도 존재합니다.

    - 생성자 방식보다 느림
    - 보일러 플레이트 코드 증가

  ### Spring 개발에서는…

  Lombok이라는 유틸리티 라이브러리를 통하여 어노테이션 기반으로 객체의 빌더를 생성할 수 있게 합니다.

    ```java
    @Builder
    public class Human {
    	
    	private String name;
    	private int age;
    	private double height;
    	private double weight;
    	
    	/**
    	 *   같은 클래스 내에서 빌더 어노테이션을 2개 이상 선언할 시,
    	 *   builder 메소드의 이름을 명시적으로 지정해주어야 한다!!
    	 */
    	@Builder(builderMethodName = "noHeightWeightBuilder")
    	public Human(String name, int age) {
    		this.name;
    		this.age;
    	}
    ```

- record vs static class

    <aside>
    🚧

  ### 깨끗한 구조를 위한 객체 단위 의도 명시

    </aside>

  `static` 클래스는 독립적으로 작용하지만, 파일 단위로 선언할 수 없으며, 항상 특정 클래스에 중첩되어 존재합니다. `record` 객체는 파일 단위로 선언할 수 있으며, 상속할 수 없고, 읽기 가능한 `불변 객체`입니다.

  record 객체를 사용하는 것이 더 좋은 이유는

    - 파일 구조 측면에서 Entity와 책임을 분리할 수 있다
    - `getField()` 메소드를 구현해서 필드에 접근하는 것이 아니며, `field()` 로 접근한다
        - 이는 record 클래스 자체가 지원하는 메소드로, 개발자가 구현할 필요 없다.
    - 모든 필드가 immutable이므로, JIT 최적화에 유리하여 런타임 성능이 향상될 수 있다.

  ### 타입 다형성

    </aside>

  객체가 서로 위임하는 형태 등으로 서로 데이터를 전달받을 때, 특정 객체의 필드가 변할 때가 있다.

    ```java
    import java.util.*;
    
    public class Main {
    	public static void main(String args[]) {
    		
    		// 가장 대표적인 예시는 List!
    		List<Student> studentList = new ArrayList<Student>;
    		
    	}
    }
    ```

  List, Map 등은 Java에서 자료구조를 쉽게 사용할 수 있도록 하는 라이브러리인데, 이 같은 경우 어떤 객체가 들어오더라도 자료구조를 형성할 수 있어야 하므로 타입 다형성이 필요하다.

  ### Generics

  이 문법은 클래스에 미리 이 타입은 정적으로 고정되지 않으며, 어느 타입으로든 들어올 수 있다는 명시를 해주는 문법이다.

    ```java
    public class SuccessResponse<T> {
    	private final Boolean isSuccess;
    	private final int httpStatus;
    	private final String code;
    	private final String message;
    	private final T data;
    	private final LocalDateTime timeStamp = LocalDateTime.now().format()// 생략
    }
    ```

  백엔드 서버에서 사용하는 가장 대표적인 Generics 사용처는 공통 응답 규격 객체이다.

  백엔드, 특히 API 서버는 JSON 문법으로 응답을 맞추어 보내주는 것이 대표적인데, 성공했는지 여부와 성공 시 반환할 객체, 실패했으면 어떤 이유로 실패했는지에 대해서, 그리고 현재 시각을 나타내주는 등 고정된 템플릿이 있는데, 이 data 필드는 어떤 DTO가 들어올지 모르므로, 타입 다형성을 이용하여 템플릿 패턴을 적용시키는 것이 설계적으로 깔끔하다.

- @RestControllerAdvice이란?

  `@ControllerAdvice`와 `@RestController`를 결합한 어노테이션이다.

  ### @ControllerAdvice

  Exception은 코드, 함수, 블록 내에서 발생해 호출했던 상위로 점점 타고 올라가서, 최종적으로 던져집니다.
  Spring같은 경우에 DB 관련 작업이나, 서비스 로직 상에서 잘못된 요청 등을 처리하기 위해서 개발자가 선언한 예외를 사용할 수 있습니다.
  이 예외는 자신을 호출한 컨트롤러까지 올라가고 나서, 예외를 수집하여 왜 이런 예외가 발생했는지, 어떻게 해결할 수 있는지 등을 모델에 작성해 뷰로 전달해야 합니다.

  `@ControllerAdvice` 는 모든 컨트롤러에서 던져진 예외에 대해 정해진 예외 처리 메소드가 있는지 탐색할 수 있도록 클래스에 수식하는 어노테이션입니다.

    ```java
    @ControllerAdvice
    public class GlobalExceptionHandler {
    	
    	// 특정 예외에 대한 처리는 @ExceptionHandler 어노테이션으로 수식!
    	@ExceptionHandler(MemberNotFoundException.class)
        public ResponseEntity<ApiResponse<Void>> handleMemberException(
                ProjectException e
        ) {
            BaseErrorCode errorCode = e.getErrorCode();
            return ResponseEntity.status(errorCode.getStatus())
                    .body(ApiResponse.onFailure(errorCode, null));
        }
    ```

- Optional이란?

    <aside>
    🚫

  Null-Safe

    </aside>

  `Null` 값일 수도 있고, 아닐 수도 있는 래퍼 클래스입니다. `isPresent()` 로 값이 존재하는지 확인할 수 있으며, `get()` 로 값을 얻을 수 있습니다.

  값이 존재하냐, 존재하지 않느냐에 따라 부가적인 메소드가 제공됩니다.

  `orElse()` 는 값이 존재할 경우 리턴, 값이 존재하지 않을 경우는 특정 로직을 실행시킬 수 있고, `ifPresent()` 는 값이 존재할 때 특정 로직을 실행시킵니다.

  Spring에서는 DB에서 레코드를 조회할 때 Optional<Entity>로 조회하고, found.orElseThrow() 메소드로 커스텀 예외를 던지도록 합니다.
