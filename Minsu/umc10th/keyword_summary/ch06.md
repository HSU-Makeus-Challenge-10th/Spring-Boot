## JPA란?

JPA는 Java 애플리케이션에서 객체와 관계형 데이터베이스 테이블을 매핑하기 위한 표준 인터페이스이다.  
개발자는 SQL을 직접 많이 작성하기보다 엔티티 객체를 저장, 조회, 수정, 삭제하는 방식으로 DB를 다룰 수 있다.  
Spring Data JPA는 JPA를 더 편하게 사용하도록 Repository 추상화와 기본 CRUD 메서드를 제공한다.  
JPA 자체는 구현체가 아니며, 실제 동작은 Hibernate 같은 구현체가 담당한다.

## N+1 문제란?

N+1 문제는 처음 1번의 조회로 가져온 N개의 데이터 각각에 대해 추가 쿼리가 N번 더 실행되는 문제이다.  
예를 들어 리뷰 목록 10개를 조회한 뒤 각 리뷰의 작성자나 이미지를 지연 로딩하면 추가 SELECT가 반복될 수 있다.  
데이터가 적을 때는 티가 나지 않지만, 목록 크기가 커지면 쿼리 수가 급격히 늘어 성능 문제가 된다.  
Fetch Join, EntityGraph, IN 쿼리 등을 사용해 필요한 연관 데이터를 한 번에 가져오는 방식으로 줄일 수 있다.

## 지연로딩과 즉시로딩의 차이는?

지연로딩은 연관된 엔티티를 실제로 사용할 때 DB에서 조회하는 방식이다.  
즉시로딩은 엔티티를 조회하는 시점에 연관 엔티티까지 함께 조회하는 방식이다.  
일반적으로 JPA에서는 연관관계를 기본적으로 지연로딩으로 두고, 필요한 조회에서만 Fetch Join으로 가져오는 습관이 좋다.  
즉시로딩은 예상하지 못한 쿼리나 N+1 문제를 만들 수 있어 신중하게 사용해야 한다.

## JPQL란?

JPQL은 테이블이 아니라 엔티티 객체를 대상으로 작성하는 JPA 전용 쿼리 언어이다.  
SQL처럼 보이지만 `SELECT r FROM Review r`처럼 DB 테이블명이 아니라 엔티티 클래스명과 필드명을 사용한다.  
Spring Data JPA에서는 `@Query`를 사용해 Repository 메서드에 JPQL을 직접 작성할 수 있다.  
복잡한 조건 조회나 Fetch Join이 필요할 때 기본 메서드 이름 쿼리보다 명확하게 의도를 표현할 수 있다.

## Fetch Join란?

Fetch Join은 JPQL에서 연관 엔티티를 함께 조회하도록 지시하는 문법이다.  
일반 Join은 조건 필터링에 가깝지만, Fetch Join은 조회 결과 엔티티의 연관 객체까지 영속성 컨텍스트에 채워준다.  
예를 들어 리뷰 목록을 조회할 때 작성자 정보를 함께 가져오면 리뷰마다 작성자를 조회하는 N+1 문제를 줄일 수 있다.  
다만 ToMany 컬렉션을 Fetch Join하면서 페이징하면 결과 중복과 메모리 페이징 문제가 생길 수 있어 주의해야 한다.

## @EntityGraph란?

`@EntityGraph`는 Repository 메서드에서 함께 로딩할 연관관계를 선언하는 기능이다.  
JPQL에 Fetch Join을 직접 쓰지 않아도 특정 연관 필드를 fetch 계획에 포함할 수 있다.  
간단한 연관 엔티티 로딩에는 코드가 깔끔하지만, 복잡한 조건이나 여러 Join 제어가 필요하면 JPQL Fetch Join이 더 명확할 수 있다.  
결국 목적은 지연로딩으로 인한 불필요한 추가 쿼리를 줄이는 것이다.

## commit과 flush 차이점은?

flush는 영속성 컨텍스트의 변경 내용을 DB에 SQL로 반영하는 동작이다.  
commit은 트랜잭션을 확정하는 동작이며, 보통 commit 직전에 flush가 자동으로 실행된다.  
flush가 실행되어 UPDATE나 INSERT SQL이 나갔다고 해서 트랜잭션이 확정된 것은 아니며, rollback되면 반영도 취소된다.  
Dirty Checking은 트랜잭션 안에서 엔티티 값을 바꾸면 flush 시점에 변경 감지를 통해 UPDATE SQL이 만들어지는 방식이다.
